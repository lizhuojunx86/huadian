# 01 风险与早期决策清单 v2

> 本文件是对架构文档 v1 的第一次系统性补课。
> v1 文档在数据模型、身份建模、史源冲突、隐喻引用、历法、地理模糊性、认识论分层、演进锁定等方面存在**一旦进入开发就很难回头**的风险。
> 本文逐项列出，并给出早期决策。每个决策在进入实施前，应经由首席架构师 + 古籍专家联合签字（通过 ADR）。
>
> 原则：**基础架构不得因这些已知问题在 Phase 1 之后出现破坏性变更。**

---

## A. 实体识别与身份建模

### A-1 一个人多个名号
**现象**：苏轼 6+ 号（东坡居士、雪堂、铁冠道人、坡仙…），文人常见。"室名""斋号"也视同号。
**v1 问题**：`persons` 扁平字段 `name_art TEXT` 只能存一个。
**决策**：
- 新增 `person_names` 表：`(person_id, name, name_type, start_year, end_year, source_evidence_id)`
- `name_type` 枚举：`primary / courtesy(字) / art(号) / studio(室名/斋号) / posthumous(谥号) / temple(庙号) / nickname(外号) / self_ref(自称) / alias(别称)`
- `persons` 表只保留一个 `name_primary` 和检索用的 `all_names_tsv` 派生列

### A-2 同名异人
**现象**：韩信（淮阴侯）vs 韩王信；多个范仲淹；多个张飞；唐代 N 个"李白"。
**v1 问题**：仅靠 `name_primary` 唯一性无法区分。
**决策**：
- 主键 UUID 不变
- 额外要求 `slug`：规则 `name_pinyin-dynasty-disambiguator`（如 `han-xin-western-han-huaiyin-hou`）
- 建立 `disambiguation_seeds` 种子表：收录已知同名异人清单，ingest 时强制对照
- 管线抽取时遇疑似同名必须带 `book_context + paragraph_context` 打进消歧队列

### A-3 异名同指（含称谓与标题性名号）
**现象**："高祖"在《史记》指刘邦，在《宋史》指赵匡胤；"太祖"更乱。"先帝""上""陛下"完全依赖上下文。
**决策**：
- 称谓性词汇（高祖 / 太祖 / 先帝 / 陛下 / 卿 / 妾…）列入 `role_appellations` 字典
- 抽取时必须解析到具体人物，失败则入 `unresolved_mentions` 队列
- 消歧规则：`(book_context, dynasty_context, preceding_emperor_chain)` 三维定位

### A-4 身份假说（历史人物本体不确定）
**现象**：老子 = 李耳？老莱子？太史儋？多说并存；西施的真实性存疑。
**v1 问题**：`persons` 表假设每人一个 ID。
**决策**：
- 新增 `identity_hypotheses` 表：`(canonical_person_id, hypothesis_person_id, relation_type, scholarly_support, evidence_ids, accepted_by_default)`
- `relation_type`: `same_person / possibly_same / conflated / distinct_despite_similar_name`
- 前端人物页默认按 `accepted_by_default=true` 的主假说展示，另显示"学界分歧"折叠区

### A-5 匿名 / 官职 / 代称人物
**现象**："项王曰""太守""大夫""吾友某"；只有官职没有名字。
**决策**：
- `mentions` 表（见 E-1）支持 `lemma_type = anonymous / title_only / unresolved`
- 提供后期人工 / LLM 回溯消歧接口，不强求在 ingest 时解决

### A-6 虚实混杂（传说 / 神话 / 寓言 / 复合人物）
**现象**：皇帝出生时吞流星、嫦娥、黄帝、哪吒、神农；屈原的作品中虚构人物"山鬼"。
**决策**：
- `persons.reality_status`：`historical / legendary / mythical / fictional / composite / uncertain`
- 商业化时可按 `reality_status` 做功能分流（学术版默认过滤 mythical）

## B. 时间与历法

### B-1 农历 ↔ 公历换算不可精确
**决策**：
- `HistoricalDate` 保留原始字段不丢失：`lunar_month / lunar_day / reign_era / reign_year / sexagenary_year / original_text`
- 公历字段以 `gregorian_year_min / gregorian_year_max` 区间存储
- `precision` 枚举：`day / month / year / year_range / decade / era / unknown`

### B-2 多政权并立年号
**现象**：南北朝、五代十国、辽宋西夏金同一公历年数个年号。
**决策**：
- 新增 `polities` 表（政权）
- `reign_eras` 表关联 `polity_id`；事件可同时关联多个 `reign_era`
- 事件记录"首发政权视角"+"其他政权年号"

### B-3 不确定时间
**决策**：`(year_start, year_end, precision)` 三元组，禁止单值存储。查询层按区间做交集。

### B-4 史前纪年
**决策**：`precision = 'era_only'` 或 `'unknown'`；`era` 字段记"传说时代 / 夏初 / ..."

### B-5 季节与相对时点
**现象**："孟春""既望""端阳""上元"指代相对时点。
**决策**：保留 `original_text_date`；另维护 `relative_date_dictionary` 将其映射为月/日区间。

## C. 文本与版本

### C-1 异体字 / 繁简
**决策**：
- ingest 时统一存"标准繁体"为底版 + 自动派生简体搜索索引
- 双向索引：搜简得繁、搜繁得简

### C-2 通假字
**决策**：建立 `variant_chars` 字典（`char, variant, context_rule`），检索层按字典扩展查询。

### C-3 避讳字
**决策**：`taboo_chars` 规则表（`dynasty, original_char, substitute_char`）；抽取层做双向补偿。

### C-4 多版本文本（中华书局 / 四库 / 百衲本）
**决策**：
- `raw_texts` 保留 `text_version` 字段；同一段落允许多版本并存
- 新增 `text_variants` 表：`(raw_text_id_a, raw_text_id_b, diff_position, diff_type, diff_content)`
- 默认检索使用"最可靠版本"（由 Book 元数据 `authoritative_version` 决定）

### C-5 校勘注 / 前人批注
**决策**：新增 `textual_notes` 表：`(raw_text_id, position, note_type, commentator, content)`；`note_type`: `emendation / annotation / parallelism / commentary`

### C-6 OCR 错误
**决策**：`raw_texts.metadata.ocr_confidence`；低于阈值的段落默认不参与抽取，进入人工校对队列

## D. 关系与因果

### D-1 不对称关系
**决策**：`relationships.direction` 严格启用；`relation_type` 枚举表中明确每种类型的方向语义。师生、君臣禁止 `bidirectional`。

### D-2 时段性关系
**决策**：`start_year / end_year` 必填（可带不确定区间）；新增 `status = active / ended / unknown`。

### D-3 关系矛盾
**决策**：允许同 `(person_a, person_b, relation_type)` 多条记录并存，每条挂不同 `source_evidence_id` 与 `account_id`；通过 `controversy_groups` 表聚合。

### D-4 关系强度
**决策**：`strength REAL (0-1)`，由共现事件数 + 文本共现度 + 人工权重合成；展示层用于图的边粗细。

## E. 引用、隐喻、化用（用户特别关注）

### E-1 显式 vs 隐式提及
**现象**："不可沽名学霸王"未提项羽名字，但所有人都知道指他；"冯唐易老"没提冯唐具体事迹但暗含典故。
**v1 问题**：`evidence_links` 只能表达"此证据支撑此实体"，不能区分显式/隐式。
**决策**：
- 新增 `mentions` 表（独立于 evidence_links，粒度更细）：
  ```
  mentions(
    id, raw_text_id, position_start, position_end,
    surface_text,              -- 原文表面形式 e.g. "霸王"
    entity_type, entity_id,    -- 指向谁
    mention_type,              -- explicit / implicit / allusion / metaphor / pun / pseudonym
    confidence,                -- 0-1
    inference_basis,           -- 推断依据（rule / context / llm / human）
    detector_version,          -- 抽取器版本
    human_verified
  )
  ```
- `evidence_links` 保留用于"实体→证据"的聚合视图；`mentions` 是位置级提及

### E-2 典故化用
**决策**：
- `allusion_usages(allusion_id, raw_text_id, mention_id, semantic_shift)` 记录每次化用
- `semantic_shift` 枚举：`faithful / extended / ironic / subverted`

### E-3 互文 / 化句 / 点铁成金
**决策**：`intertextual_links(source_raw_text_id, target_raw_text_id, link_type, overlap_tokens, confidence)`
`link_type`: `quote / paraphrase / rebuttal / imitation / parody`

### E-4 诗中之地、诗中之人（隐指）
**现象**：王昌龄"秦时明月汉时关"中"关"不是某一具体关隘；"西施"在后世诗中常是"美人"的代称。
**决策**：隐指人物用 `mentions.mention_type = metaphor`，不强制 resolve 到具体实体

## F. 史源冲突建模（最重要的架构决策之一）

### F-1 "单一事实"是幻觉
**v1 问题**：`events` 表假设鸿门宴只有一个事实版本。但《史记》《汉书》记载细节差异大，《资治通鉴》又做取舍。
**决策**：**将 `events` 拆成两层**：
```
events(id, title, canonical_account_id, ...)     -- 事件抽象锚（集合）
event_accounts(
  id, event_id, source_book,
  narrative,                 -- 该叙述的具体内容
  participants, places, sequence,
  source_evidence_ids,
  is_canonical,              -- 是否默认展示
  reliability_score          -- 综合可信度
)
```
- Event 是"鸿门宴这个指代"，Account 是"《史记·项羽本纪》的鸿门宴叙述"
- 前端默认展示 canonical Account；用户可切换或并排对比
- `version_conflicts` 升级为 `account_conflicts`，在 Account 间度量差异

### F-2 Account 自动聚类
**决策**：相似度 + 实体共指 → 聚类 → 人工合并；新 Account 默认不自动合并

### F-3 来源可信度分级
**决策**：
- `books` 表新增 `credibility_tier`：`primary_official(正史) / primary_unofficial(野史笔记) / secondary_compilation(编撰) / tertiary_reference(工具书) / folk(民间传说)`
- `source_evidences.provenance_tier`：`primary_text / scholarly_consensus / ai_inference / community_contribution / expert_annotation`

## G. 地理空间

### G-1 地名是点还是域
**决策**：`places.geometry GEOMETRY(Geometry, 4326)`（支持 Point / LineString / Polygon）+ `fuzziness REAL`（0=精确点，1=模糊区域如"江东"）

### G-2 同名不同时疆域
**决策**：`place_names` 扩展 `geometry_variant GEOMETRY`，存该朝代的地理范围

### G-3 行政区划演变
**决策**：`place_hierarchies(place_id, parent_place_id, dynasty, admin_level)` 表达"A 朝代属 B 郡"

### G-4 CHGIS 覆盖不全
**决策**：
- 优先 CHGIS 匹配
- 未命中：LLM 推断 + `geocoding_confidence < 0.5` 入人工复核
- 未命中地名不阻塞入库，仅标记 `geocoded = false`

### G-5 跨境地名
**决策**：`places.modern_country`（默认 CN，可为 KR/JP/VN/MN/RU 等），不限定中国疆域

### G-6 虚构地名（"桃花源""瑶池"）
**决策**：`places.reality_status` 与人物对齐（`historical/legendary/mythical/fictional`）

---

## H. 认识论与溯源分层

### H-1 事实来源层级
**决策**：所有实体字段隐式带 `provenance_tier`，五级（见 F-3）。前端用徽标展示：
- 🏛 来自原文
- 📖 学界共识
- 🤖 AI 推断
- 👥 众包贡献
- 🎓 专家标注

### H-2 修订历史不可丢
**决策**：所有主表启用 `entity_revisions`：`(entity_type, entity_id, revision_no, diff, actor, reason, created_at)`

### H-3 不可物理删除
**决策**：统一 `deleted_at` soft-delete；审计保留 ≥ 7 年

## I. 国际化与商业化

### I-1 字段多语言
**决策**：所有 user-facing TEXT 改为 JSONB 多语言结构：
```json
{"zh-Hans": "...", "zh-Hant": "...", "en": "...", "ja": "..."}
```
检索层按请求头 `Accept-Language` 路由；底层至少存 `zh-Hans`。

### I-2 许可证与版权
**决策**：`books` 表必填 `license`（CC0 / CC-BY / public_domain / proprietary / unknown）；proprietary 不提供全文导出 API

### I-3 API 计费预留
**决策**：API Gateway 从一开始预留 `tenant_id / api_key / usage_meter`，即便 MVP 阶段默认 public

### I-4 数据分区
**决策**：大表按 `book_title` 或 `dynasty` 分区，便于未来切分

### I-5 隐私与合规
**决策**：
- 用户输入（反馈、注释、模拟器对话）必须与实体数据隔离存储
- 预留 GDPR-style 数据导出 / 删除接口
- 未成年用户界面预留（教育版可能需要）

## J. 演进与技术锁定

### J-1 URL 稳定性
**决策**：所有实体强制 `slug`（唯一、不变、人类可读）；API 和前端优先使用 slug

### J-2 GraphQL breaking change
**决策**：
- 严格使用 `@deprecated` 标注；新增字段 > 修改字段 > 删除字段
- schema registry 工具（如 Apollo Studio 或自建）追踪 breaking change

### J-3 Embedding 模型切换
**决策**：单字段 `embedding` 改为多槽位表：
```
entity_embeddings(
  entity_type, entity_id,
  model_name, model_version, dims,
  vector, created_at,
  PRIMARY KEY (entity_type, entity_id, model_name)
)
```
查询层按"当前默认模型"路由；切换模型只需回填，不破坏旧数据

### J-4 Prompt 迭代
**决策**：
- 每个 prompt 模板有 `prompt_id + version + hash`
- LLM 响应按 `(prompt_hash, input_hash, model)` 缓存
- `extractions_history` 表保留所有版本的抽取结果，允许回滚对比

### J-5 单 PG 扩展触发器
**决策**：进入切出评估的**具体指标**：
- pgvector 向量数 > 50M
- 图遍历 p95 > 2s（3 度以内）
- 全文索引重建 > 6 小时
- DB 存储 > 2 TB
任一触发，启动 RFC 讨论是否切出模块数据库

## K. 质检与可观测性

### K-1 Confidence 必须校准
**决策**：禁止直接返回 LLM 自报的 confidence；建立校准数据集，定期用 isotonic regression 校准

### K-2 管线断点续跑
**决策**：
- 任务记录 `(book, volume, paragraph, step, status, prompt_version, worker_id)`
- 任务幂等 key：`(paragraph_id, step, prompt_version)`，重跑 skip 已完成

### K-3 前端错误回溯
**决策**：
- 所有 API 响应带 `trace_id`
- 前端错误（Sentry）与 `trace_id` 关联
- 用户"存疑"按钮生成 `feedback` 记录：`(user_id, entity_type, entity_id, evidence_id, prompt_version, model_version, note)`

### K-4 TraceGuard 集成（详见 docs/06）
**决策**：将 TraceGuard 确立为管线 checkpoint 的**运行时底座**；在每个 LLM 步骤后插入 TraceGuard checkpoint，失败自动触发重试 / 降级 / 人工队列

## L. 团队协作与流程

### L-1 单 agent 过载与越界
**决策**：`.claude/agents/{role}.md` 明确每个角色边界；根 `CLAUDE.md` 强制"开工三步"；跨职越位视为违规

### L-2 知识断续
**决策**：`docs/STATUS.md + docs/CHANGELOG.md + docs/decisions/ADR-*.md + docs/tasks/T-*.md` 四件套维持项目持续可接续

### L-3 LLM 窗口限制
**决策**：每个 agent 任务输入 ≤ 30% 窗口；超出需拆任务或调用子 agent；禁止把整个项目代码库一次性喂入

---

## M. 可用性与交互（架构师需提前锁定的）

### M-1 古今搜索双模
**决策**：搜索框同时接受现代名（"西安"）和古名（"长安"），后端双向路由

### M-2 时间维度贯穿全站
**决策**：时间滑块不是"地图的功能"，是"全局过滤器"，URL 参数 `?year=742&precision=day` 全站生效

### M-3 溯源 UI 强制可见
**决策**：任何结论性展示旁必须有"查看原文证据"入口，不可隐藏 ≥ 2 级菜单

## 本清单的使用方式

1. Phase 0 启动前，PM + 历史专家 + 架构师联合评审本清单，将每条决策确认入 ADR
2. 未决项（U-01 ~ U-07）在相应 Phase 开始前必须有结论
3. 任何后续发现的新风险点，增补到本清单并升版（v3, v4...），每次升版记入 CHANGELOG
4. 开发期间若发现某条决策不适用，不得偷偷更改实现，必须回到本文件修订并记 ADR
