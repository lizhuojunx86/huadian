> 触发：用户批准 B3「L1 解冻」（同意架构师推荐的"下沉 huadian_classics 专有逻辑到 examples，把领域无关比例从 ~58% 顶到 70%"）。

## 1. 一句话结论

**core 已经是干净的（0 领域字样），没有"领域泄漏"可下沉。** 领域无关比例 64.4% 偏低，不是工程缺陷，而是"框架只有一个很重的 example（史记）"造成的度量假象。要真正把比例顶到 70%，靠的不是搬代码，而是**①给 import root 改名（ADR-037 已主动 defer，需解冻）**或**②引入第 2 个真实 domain**。两者都需要你拍板，故先停下来报告。

## 2. 测量（2026-06-20 实测）

```
core   (领域无关)：4661 行   ← framework/*/ 主模块 .py（不含 tests / examples）
tests  (领域无关)：1819 行   ← framework/*/tests/
examples/huadian_classics (领域专属)：3578 行
总计：10058 行

领域无关比例 = (4661 + 1819) / 10058 = 64.4%
（若只算 core：4661 / 10058 = 46.3%）
```

D-route §10 目标 ≥ 70%。当前 64.4%，差 ~5.6 个百分点。

> 注：2026-06-07 进度评估记的 "~56–62%" 与本次 64.4% 的差异来自口径（是否计入 tests、是否计入非 .py）。本次统一口径为"core+tests 占全部 .py 行"。

## 3. 核查：原方案"下沉泄漏"为何不成立

原方案假设 core 里混入了领域专属逻辑，搬到 examples 即可提升比例。核查否定了这个假设：

**3.1 core 已干净**：与 [d-route-progress-review-2026-06-07](d-route-progress-review-2026-06-07.md) §4.2 一致——core 与 examples 已用 PEP 544 Protocol 工程隔离，core 文件 0 中药/史记字样。没有泄漏可下沉。

**3.2 "低关键词密度"嫌疑文件逐个证伪**：反向找"是否有通用代码错放在 example 里、可上提到 core"。3 个按关键词看密度低的文件，逐文件核查后确认**全部真·领域专属**：

| 文件 | 行数 | 初看 | 核查结论 |
|------|------|------|---------|
| `audit_triage/examples/.../asyncpg_store.py` | 380 | 像通用 Postgres adapter | SQL 直接引用 `person_names` / `person_id` / `target_entity_type='person'` 史记表结构 → **领域专属** |
| `invariant_scaffold/examples/.../runner_setup.py` | 65 | 像通用 runner 装配 | import 的是 `invariants_v4/v6/v8/v9/v10/v11` 史记不变量 → **领域专属** |
| `invariant_scaffold/examples/.../db_port.py` | 97 | 像通用 DBPort | 是该 example runner 的 asyncpg 绑定，服务于上面两个 → **领域专属** |

关键词密度低只是因为 grep 没匹配到 SQL 表名和 import 名，不代表通用。

**3.3 推论**：examples/huadian_classics 的 3578 行是**合法的、真实的 dogfood 资产**（byte-identical 等价测试 + 生产 adapter + 完整 invariant 集）。删它/砍它 = 丢掉框架可移植性的活体证明。所以**靠动 examples 来刷比例 = 损坏资产换数字，是反模式**（违背项目"不做完成度剧场"的纪律）。

## 4. 真正的杠杆（需要你决策）

"领域无关比例 ≥ 70%"这个判据，对一个**只有一个 example 的框架**天然难达标——任何"框架 + 一个丰富案例"的仓库，按 LOC 算都会显得"领域偏重"。这不是 bug，是单案例阶段的结构性特征。真正能动它、且**同时真的改进框架**的只有两条路：

### 选项 ①：import-root 改名 `framework.*` → `kb_forge.*`
- **价值**：这是 v0.1 release 的**真实硬门槛**（ADR-037 已把命名定为 `kb-forge`，但 import root 改名"延到 v0.1 前专门 sprint 避免 churn"）。做了它，L1 才算真正"解冻"且向 v0.1 推进。
- **对 70% 比例**：改名本身不直接提升比例（只是改命名空间），但它是把框架从"huadian 内部子目录"变成"独立可发布包"的关键一步，是比"刷比例"更重要的 framework-ness 信号。
- **代价/风险**：跨全部 framework 文件的机械 churn；需核查仓内/app 是否有外部 `import framework.*`；与 ADR-037 的 defer 决策冲突 → **需要你批准解冻 ADR-037**。
- **红线**：批量重命名属 CLAUDE.md §5 红线 #3，执行前我会出完整旧→新映射表 + 影响面 + 回滚方案再确认。

### 选项 ②：引入第 2 个真实 domain（非史记、非中药叙事域）
- **价值**：这才是"领域无关"判据**真正想验证的东西**——框架能不能被第二个无关领域复用。traceguard 的 tcm_extraction PoC（GO-Z-η）已是半个证明；一个完整的第 2 domain example 会是硬证明。
- **对 70% 比例**：第 2 example 也是领域专属 LOC，**不直接提升比例**——但它把"比例"这个代理指标的意义补上了（从"单案例"变"多案例"）。
- **代价**：是一个完整 sprint 的工作量，需要选域 + 数据。

### 选项 ③：重定义判据（承认 64.4% 在单案例阶段是合理的）
- 把"领域无关比例 ≥ 70%"改判据为"**core+tests 0 领域字样**"（已达成 ✅）+"**≥2 个 domain example 各自隔离**"（待选项 ② 达成）。即用"隔离是否干净 + 是否多域"替代纯 LOC 比例。
- **价值**：去掉一个对单案例阶段有 Goodhart 风险的 KPI；**风险**：可能被读作"为达标而改标准"，需在 ADR 里讲清理由。

---

## 5. 推荐

**短期推荐选项 ①（import-root 改名）+ 选项 ③（重定义判据）组合**：
- ① 是 v0.1 的真门槛，且是"真解冻 L1"最实的一步；
- ③ 把"70% LOC"这个对单案例阶段失真的指标，换成"隔离干净 + 多域"这种更直接测 framework-ness 的判据；
- 选项 ②（第 2 domain）作为 2027 路线图里"验证可移植性"的正式里程碑，不在本轮硬塞。

但 ① 与 ADR-037 的 defer 冲突，③ 改判据属"对工作计划的实质改动"——按项目纪律（CLAUDE.md §5 红线 + 项目指令"实质改动需用户批准"），**两者都要你点头我才动**。这也是我没有直接执行 B3、而是先出本报告的原因。
