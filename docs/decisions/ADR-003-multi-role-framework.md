# ADR-003: 多角色 Agent 协作框架启用

- **状态**：accepted
- **日期**：2026-04-15
- **提议人**：首席架构师
- **决策人**：用户（原始诉求）+ 首席架构师（授权落地）
- **影响范围**：整个开发流程、`.claude/agents/` 目录、任务卡制度、跨会话接续机制
- **Supersedes**：无
- **Superseded by**：无

## 背景

用户编程背景较弱，但要求：
1. 开发质量达到中型开发团队水准
2. 各专业领域不得互相越位（前端不得自行决定视觉；管线不得自行决定实体定义；等等）
3. 跨会话 LLM 上下文窗口限制下仍能稳定推进

单一"全能 agent"会出现：
- 跨领域决策被默默压缩（例如前端 agent 顺手改了 schema）
- 上下文膨胀，质量劣化
- 没有评审环节，风险点被掩盖

## 选项

### 选项 A：单 agent + TodoList 自律
- 优点：简单
- 缺点：无法强制解耦；已验证的风险会被忽略

### 选项 B：多角色 agent（Claude Code sub-agent 机制）+ 角色文件 + 任务卡制度
- 优点：每个 agent 有清晰边界；跨角色交接有契约；任务卡保证可接续
- 缺点：学习成本；agent 间协调需要规则

### 选项 C：人工团队 + Claude 仅作辅助
- 优点：真实团队质量
- 缺点：不符合项目定位（用户单人开发）

## 决策

**选择 B：启用 10 角色 agent 框架**。

### 10 角色（已在 `.claude/agents/` 落地）

| # | 角色 | 文件 | 模型 |
|---|------|------|------|
| 1 | 首席架构师 | `chief-architect.md` | opus |
| 2 | 产品经理 | `product-manager.md` | opus |
| 3 | 古籍/历史专家 | `historian.md` | opus |
| 4 | UI/UX 设计师 | `ui-ux-designer.md` | opus |
| 5 | 后端工程师 | `backend-engineer.md` | sonnet |
| 6 | 数据管线工程师 | `pipeline-engineer.md` | sonnet |
| 7 | 前端工程师 | `frontend-engineer.md` | sonnet |
| 8 | QA 工程师 | `qa-engineer.md` | sonnet |
| 9 | DevOps / SRE | `devops-engineer.md` | sonnet |
| 10 | 数据分析师 | `data-analyst.md` | sonnet |

### 启用规则

1. **入口约束**：每个 agent 启动必须读 CLAUDE.md + STATUS.md + CHANGELOG.md 最近 5 条 + 本角色文件 + 任务卡
2. **任务卡主导角色**：每张任务卡声明"主导角色 + 协作角色"，只有主导角色能动主产出
3. **禁区硬约束**：每个角色文件列出"❌ 不得跨越"的事项；触禁区必须报架构师仲裁
4. **冲突升级**：任意两角色冲突 → 架构师仲裁 → 记入 ADR

### RACI 简表（示例）

| 工作项 | R (责任) | A (批准) | C (咨询) | I (知情) |
|-------|---------|---------|---------|---------|
| GraphQL schema | 后端 | 架构师 | 前端/管线 | 全体 |
| Prompt 模板 | 管线 | 历史专家 | 架构师 | QA |
| 视觉风格 | 设计师 | 设计师 | PM | 前端 |
| 实体定义 | 历史专家 | 历史专家 | 架构师 | 管线/后端 |
| 部署架构 | DevOps | 架构师 | 后端 | 全体 |

完整 RACI 见 `docs/03_多角色协作框架.md`。

## 影响

- 正面：专业解耦；跨会话可接续；每次决策有迹可循
- 负面：任务卡/ADR 维护开销；启动入口约束要求每个 agent 多读 3~5 个文件
- 迁移成本：0（起始态）

## 回滚方案

若实践中发现多角色协作开销过大，允许合并部分角色（例如 DevOps + 后端合并）。但**架构师 / 历史专家 / 设计师 / PM** 四角色不得合并，因这是专业判断的关键分界线。
