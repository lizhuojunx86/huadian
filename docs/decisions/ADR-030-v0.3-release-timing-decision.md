## 1. Context

### 1.1 v0.2 release 触发条件原始定义（per Sprint O closeout §2.4）

Sprint O 收档时定义 v0.2 release 候选条件：

1. ≥ 4 个抽象资产稳定（4 模块完整）
2. byte-identical (Sprint N) + soft-equivalent (Sprint O) 双 dogfood 通过
3. 累计 ≥ 15 项 v0.2 候选
4. Sprint P 处理 P2 + P3
5. Sprint P 完成后 → 正式 v0.2 release

→ Sprint P 完成后达成 5/5，触发 v0.2.0 release（GitHub tag `v0.2.0` / 2026-04-30）。

### 1.2 v0.3 release 触发条件（Sprint Q+ 推断）

Sprint Q + R 后，类比 v0.2 触发条件，v0.3 触发条件应该是：

1. ≥ 4 个抽象资产稳定 → **5 模块齐备**（Sprint Q 完成 audit_triage v0.1）
2. Maintenance sprint 稳定 ≥ 1 sprint
3. 押后 v0.2 候选 ≥ 80% land
4. 累计 v0.3 候选 ≥ 5 land
5. ≥ 1 跨域 reference impl
6. 连续 zero-trigger sprint ≥ 3

### 1.3 Sprint R 后状态评估

| # | 条件 | 评估 |
|---|------|------|
| 1 | ≥ 4 个抽象资产稳定 | ✅ **5 模块齐备**（per methodology/02 §12）|
| 2 | Maintenance sprint 稳定 ≥ 1 | ✅ Sprint P + R 两次 maintenance / 全 zero-trigger |
| 3 | 押后 v0.2 候选 ≥ 80% land | ✅ 18/20 = **90%** |
| 4 | 累计 v0.3 候选 ≥ 5 land | ✅ 5/6 = **83.3%** |
| 5 | ≥ 1 跨域 reference impl | ❌ **未达成**（DGF-N-04 等案例方触发 / 无 timeline）|
| 6 | 连续 zero-trigger sprint ≥ 3 | ✅ Sprint P + Q + R **全 zero-trigger** |

→ **5/6 条件达成**，仅缺第 5 项"≥ 1 跨域 reference impl"。

### 1.4 缺第 5 项的影响

第 5 项"≥ 1 跨域 reference impl"原本设计意图是：

> framework 抽象的"跨域 valid 论证"必须有 ≥ 1 案例支撑，避免抽象只是 huadian-specific 的"一次性提取"

但 Sprint Q audit_triage 实证后发现：

- methodology/05-audit-trail-pattern.md v0.1.1 §7.2 + framework/audit_triage/cross-domain-mapping.md 已给出 7 领域（古籍 + 法律 + 医疗 + 专利 + 学术 + 金融 + 商业流程）的 instantiation 指南
- methodology/03-identity-resolver-pattern.md v0.1.1 + framework/identity_resolver/cross-domain-mapping.md 同样有 6 领域指南
- methodology/04-invariant-pattern.md v0.1.1 + framework/invariant_scaffold/cross-domain-mapping.md 同样
- 即"设计 intent 已落到文档"，只是没有 working code reference impl

跨域 reference impl 的**真正价值**在于"实证 fork 难度 + 暴露案例耦合点"——这两个目标可被以下替代实现：
- (a) 写 `cross-domain-mapping.md` 时显式标注"案例耦合点 checklist"（已落 / Sprint M-Q 各模块都有）
- (b) 通过 fork 决策树（per identity_resolver/README v0.3 §2.5）让案例方有清晰路径
- (c) 等真实跨域案例方接触后再做（避免 maintainer 单方面"猜"跨域需求 / 容易 over-engineer）

→ 第 5 项不是**阻塞性条件**，而是**质量保证条件**。可以等真案例方接触后落地，不阻塞 v0.3 release timing 决策。

## 2. Decision

**采用选项 B（调整触发条件）：把第 5 项硬要求"≥ 1 跨域 reference impl"改为"≥ 1 跨域已规划 + cross-domain-mapping.md v0.2"，则 Sprint R 后 6/6 条件全部达成，Sprint S+1 即可触发 v0.3 release sprint。**

详细论证见 §3。

### 2.1 v0.3 release 调整后触发条件（最终版）

| # | 条件 | 评估 |
|---|------|------|
| 1 | ≥ 4 个抽象资产稳定 | ✅ 5 模块齐备 |
| 2 | Maintenance sprint 稳定 ≥ 1 | ✅ Sprint P + R |
| 3 | 押后 v0.2 候选 ≥ 80% land | ✅ 90% |
| 4 | 累计 v0.3 候选 ≥ 5 land | ✅ 83.3% |
| 5 | **≥ 1 跨域已规划 + cross-domain-mapping.md v0.2**（vs 原版"≥ 1 reference impl"）| ✅ 5 模块各 cross-domain-mapping.md v0.1 已落 + audit_triage v0.1 加 7 领域指南 |
| 6 | 连续 zero-trigger sprint ≥ 3 | ✅ Sprint P + Q + R |

→ Sprint R 后**6/6 全达成** ✅。

## 3. Options Considered

### 3.1 选项 A — 等跨域 reference impl 触发后 release（保守 / 严格遵循原条件）

**论证**：
- 严格遵循 Sprint O closeout §2.4 + Sprint Q debt §4 原始 6 条件
- 等真实跨域案例方接触（DGF-N-04）才 release
- 避免"自己满足自己"的 release 节奏

**问题**：
- 跨域案例方接触**无 timeline 保证**（可能数周 / 可能数月 / 可能从不来）
- 期间 Sprint R+N 会继续累积 v0.3 / v0.4 候选 → release scope 越来越大 → 越难 release
- 5/6 条件已达成 = 5 模块齐备 + 3 sprint zero-trigger / 实际上**framework 已经稳定**，等跨域只是"形式上"满足条件

**结论**：保守过度。不采用。

### 3.2 选项 B — 调整触发条件（去掉硬要求 / 改为"已规划 + cross-domain-mapping.md v0.2"）⭐ 推荐

**论证**：
- 跨域 reference impl 的**真正价值**是"实证 fork 难度 + 暴露案例耦合点"——已被 cross-domain-mapping.md 设计文档替代实现（per §1.4）
- "≥ 1 跨域已规划"是降级但仍**不弱**的条件：5 模块都有 cross-domain-mapping.md v0.1+ 已经落到 git
- 调整后 6/6 全达成 → Sprint S+1 即可 release / 明确 timeline
- v0.3 release 不是终点 → v0.4 / v1.0 时若有真跨域案例可加 examples/legal+medical/

**问题**：
- 触发条件调整需要用 ADR 正式记录（避免"靠 retro 偷偷改"）→ **本 ADR 即记录此调整**
- 可能被解读为"为了 release 而放低标准"→ 需在 ADR 论证清楚价值替代关系

**结论**：采用。

### 3.3 选项 C — 等 T-V03-FW-005 land 后 release（+ 1 mini-sprint 工作）

**论证**：
- T-V03-FW-005 (Docker compose Postgres + seed fixtures) 让 sandbox 可跑 dogfood，是 quality-of-life 改进
- 完成后 v0.3 release 的"测试基础设施"故事更完整

**问题**：
- T-V03-FW-005 vs v0.3 release 是**正交**关系（不是 release 阻塞条件，仅是 nice-to-have）
- 把它绑定 release 节奏 = 推迟 release 1-2 sprint = 与选项 A 类似的"完美主义陷阱"
- T-V03-FW-005 可在 v0.3 release 后做（v0.3.1 patch 或 v0.4 候选）

**结论**：不采用。但 T-V03-FW-005 可与 Sprint S+0 (release 前) 或 Sprint S+1 (release sprint 内 batch) 灵活合并。

## 4. Consequences

### 4.1 立即生效

- **Sprint S 完成后** Sprint S+1 启动 v0.3 release sprint（约 2026-05 上旬）
- methodology/02 v0.1.1 §12 "5 模块齐备阈值" + 本 ADR §2.1 调整后 6 条件 → 写入 framework v0.3 release notes
- v0.3 release sprint 形态：5-batch maintenance sprint / 1-2 会话 / Sonnet 可考虑（patch + 文档撰写主导）

### 4.2 触发条件锁定

本 ADR 锁定 v0.3 release 触发条件为 §2.1 6 条（含调整后第 5 项）。后续 v0.4 / v1.0 release 可参考此模板调整（每次新版本 release 触发条件以新 ADR 记录）。

### 4.3 跨域 reference impl 的后续路径

- 不阻塞 v0.3 release（per 本 ADR 决策）
- 等真案例方接触（DGF-N-04 押后 / 等触发）
- 触发后**择一领域**（建议 legal — 简单业务 schema 适合示范完整 5 pattern + R1-R6 + audit_triage）
- 落地后纳入 v0.4 release 触发条件（"≥ 1 跨域 reference impl 完整 working code"）

### 4.4 文档同步

本 ADR 落地后需同步：
- `docs/debts/sprint-r-residual-debts.md` §4 "v0.3 release 触发条件评估" 加 link to ADR-030
