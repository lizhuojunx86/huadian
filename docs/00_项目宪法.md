# 00 项目宪法 (Project Constitution)

> 本文件定义华典智谱项目的**不可变原则**。任何后续设计、决策、代码提交都不得违背本文的条款。
> 修订本文件等同于"修宪"——必须由首席架构师提议、记入 ADR、并在 CHANGELOG 中显式公告。

## 一、数据第一性

**C-1 一次结构化，N 次衍生。**
所有产品功能均为对底层统一知识图谱的查询视图。禁止为单个产品建立独立数据表或绕过知识层。

**C-2 所有实体必须可溯源。**
任何向用户展示的事实、关系、推断都必须能回溯到至少一条 `SourceEvidence`，或清晰标注为 AI 推断 / 学界共识 / 众包贡献（provenance tier）。没有溯源的数据不得入库。

**C-3 多源共存优于单源定论。**
同一事件、同一关系在不同古籍中记载可能冲突。系统必须支持"多叙述（Account）并存"，而不是强行选出一个"真相"。

**C-4 历史原貌不可丢。**
任何数据修订必须保留原版本（`entity_revisions` 审计表）。soft-delete 是默认策略，物理删除必须经架构师审批并记入 ADR。

## 二、架构约束

**C-5 单 PostgreSQL 优先。**
在数据规模未触达"切出触发器"（见 ADR-001）前，禁止引入独立图数据库、独立向量库、独立全文搜索引擎。降低运维复杂度优先。

**C-6 Schema-first。**
任何 API / 数据结构变更，先改 Drizzle schema 或 GraphQL schema，再改实现。禁止先写代码再补 schema。

**C-7 无黑盒 LLM 调用。**
所有 LLM 调用必须通过统一 Gateway，记录 prompt 版本号、prompt hash、input hash、model、response、cost、latency。未经 Gateway 的直接 LLM 调用视为违规。

**C-8 质检嵌入，不是事后补。**
从数据层到前端层的每一层都必须内置质检 hook。质检失败的数据不得进入下游。这是 TraceGuard 在管线层的使命。

**C-9 可观测性先于性能优化。**
所有服务入库即接入 OpenTelemetry trace；前端关键交互先有埋点再谈优化。

## 三、工程约束

**C-10 类型端到端一致。**
Drizzle schema → GraphQL schema → TypeScript 类型 → 前端组件 Props，整条链路必须类型一致。`packages/shared-types` 是唯一真源。

**C-11 破坏性操作必须可回滚。**
迁移必须有 down 方案；数据处理任务必须幂等；LLM 抽取结果按 prompt 版本保留历史，不覆盖。

**C-12 多语言从一开始设计。**
所有 user-facing 文本字段（title / summary / significance / description）使用 JSONB 多语言结构。不允许事后加 i18n。

**C-13 URL 稳定。**
实体必须有人类可读 slug（唯一、不变），API 和前端优先使用 slug，UUID 仅作内部主键。

**C-14 许可证与版权前置。**
任何原始文本入库前必须标注 license 字段；商业化时按 license 过滤数据。

## 四、协作约束

**C-15 角色解耦。**
前端工程师不得决定视觉设计；管线工程师不得决定实体归类；工程师不得决定产品功能取舍；产品经理不得直接改 schema。每个角色在各自 `.claude/agents/{role}.md` 中定义的禁区不得逾越。

**C-16 决策可追溯。**
任何非琐碎的架构决策必须有对应的 ADR（`docs/decisions/ADR-NNN-*.md`）。只看代码无法理解的"为什么"必须写下来。

**C-17 进度显式化。**
任务进入执行必须有对应任务卡（`docs/tasks/T-NNN-*.md`）；完成状态通过更新 STATUS.md 和 CHANGELOG.md 显式化。口头说"完成了"不算数。

**C-18 跨会话接续友好。**
项目必须保证新的 Claude Code 会话（或新成员）在 30 分钟内能读文档就明白当前在哪、下一步做什么、为什么。

## 五、质量底线

**C-19 测试先行于优化。**
核心数据路径（实体查询、溯源、搜索、LLM 抽取）必须有回归测试；性能优化不能以牺牲正确性为代价。

**C-20 LLM 幻觉不信任。**
LLM 输出必须经过 schema 校验、证据要求、置信度评估。置信度 < 0.7 的结构化结果默认进入人工复核队列。

**C-21 用户反馈是反向管线。**
前端"存疑"按钮链路必须连通：用户质疑 → 关联 evidence_id / prompt_version / model_version → QA 看板 → 触发重抽或人工修订。这是产品"活"起来的关键。

## 六、D-route 战略原则（2026-04-29 新增）

> 本节为伴随 ADR-028 战略转型新增的不可变原则。C-22 ~ C-25 适用于 D-route 阶段（即项目当前及未来阶段），优先级与 C-1 ~ C-21 同等。如未来项目转回其他 route（A/B/C），需新 ADR + 修宪程序撤销本节。

**C-22 案例服务于框架。**
任何 sprint / 任务 / 实现工作的优先级评估，必须先回答"对框架抽象的价值"，而非"对案例完成度的价值"。纯 case-only 工作（即只对完成史记数据有用、对框架抽象无贡献）应被降级或 sunset，记入衍生债 P3 backlog。例外：当前已开工的 sprint 不溯及既往；本原则适用于 ADR-028 接受日期（2026-04-29）之后立项的工作。

**C-23 工程模式必须可抽象。**
任何新增的 invariant / guard / role / workflow / pattern / audit table 在设计时必须思考"领域无关化的路径"。即使当前不能完全抽象，至少要在 ADR / 代码注释中显式标注 `domain-specific parameters: [X, Y, Z]`，让未来抽象时知道哪些是 domain layer、哪些是 framework layer。违反此原则的工程模式不得 merge 入主分支。

**C-24 写作低频但有节律。**
D-route Layer 2（方法论文档）的产出是项目长期复利的核心资产之一。架构师**至少**每月产出 1 段决策日记（sprint retro / 决策反思 / 教训积累，不限形式但需公开 commit），**至少**每季度产出 1 篇方法论文章（可借助 AI 起草，但架构师必须 review + 公开发表）。连续 3 个月无任何 Layer 2 产出，触发 ADR-028 §5 阻塞条件 #5（架构师持续无法投入战略 / 写作），需重评 D-route 整体可行性。

**C-25 互补不竞争。**
与近邻项目（shiji-kb / 字节识典古籍 / Anthropic Cowork & SubAgent / LangChain 等）保持公开、礼貌的互补关系，不主动制造竞争叙事或对立。我们的差异化基于"做造的工具"vs"造好的数据"、"团队工程化"vs"单人英雄主义"等正面定义，不基于"我们比他们好"的话术。任何对外宣传 / 文章 / 演讲应遵循此原则。如近邻项目主动接触合作意向，架构师须以开放态度回应并记入 ADR。

## 七、修宪程序

修订本文件必须满足以下全部条件：

1. 由首席架构师提议（或经其批准）
2. 新增 ADR 说明变更原因、影响范围、回滚路径
3. 在 CHANGELOG.md 中以 `constitution:` 前缀显式记录
4. 如果变更影响数据模型或已有数据，必须同时提交迁移脚本
