# Case-2 候选评估：中成药制造全链条 AI 经营管理

> Status: **Draft v0.1 — 战略评估阶段（Stage 1 / 共 5）**
> Date: 2026-05-05
> Owner: 首席架构师 + 用户（企业 A 前负责人）
> Anchor ADR: ADR-028-strategic-pivot-to-methodology / D-route §3.4 跨领域案例机制
> 目的：评估"中成药制造全链条 AI 监测与优化建议系统"是否进入 huadian D-route Layer 3 的第二案例候选池，并给出与用户实际约束匹配的最小起点。

---

## 0. TL;DR

用户提出的"中成药全链条可视化 + 自动改进建议"愿景是**真实命题、行业级痛点**，但全量实现是 5-10 年、千万级、组织级的工程，**与用户当前可调动的资源（单人 + AI、几批 6-10 年前的批记录、不接受任何信誉风险）严重不匹配**。

直接对接现实的路径是**降维**：
- 不做"系统"，做**领域知识工程参考案例**；
- 不拉客户，先**自给自足地证明方法论可迁移**；
- 不追求覆盖全链条，先**把"提取 / 制剂"工序的质量决策框架做成可审计的小型知识资产**。

这条路与 huadian D-route Layer 3"鼓励 1-2 个跨领域第三方案例"的设计完全契合，也保留了将来"择机单点突破"到 A/B/C 任一路线的所有可能性。

## 1. 用户原始构想（保真转述）

用户作为前制药企业负责人，希望用 AI agent 团队实现中成药生产企业**全链条经营管理可视化 + 自动改进建议**。覆盖范围：

> 药材基源选择 → 种植参数 → 入场检验 → 前处理 → 提取 → 制剂 → 包装 → 仓储物流 → 销售渠道 → 患者疾病诊断适应性 → 疗效（RX 临床观察 / OTC 患者评价大数据） → 不良反应 → 产品质量稳定性 → 全链条经济性 → 全链条管理有效性 → 信息反馈准确性 / 及时性 → 数据沉淀 + 关联挖掘 → 改良型新药立项

希望该系统：
1. 自动与"标杆数据 / 企业目标数据"比对；
2. 给出可记录的改进建议；
3. 通过自有数据沉淀挖掘内部关联关系；
4. 不断改善药效、稳定质量、提高资金周转率、识别新药机会。

### 1.1 用户给出的资源约束（5 问 5 答）

| # | 问题 | 用户回答 | 关键含义 |
|---|------|---------|---------|
| 1 | 目标定位（A 单企业 / B 开源框架 / C SaaS / D 组合） | **D 组合，将来择机单点突破** | 现在不押任何具体路线，需要的是"产出可复用的资产"，不是产品 |
| 2 | 资源边界 | 用户 + Claude 等 AI | 单人 + AI 协作。无团队、无 ops、无客户支持能力 |
| 3 | 企业 A 资产 | 6-10 年前几个品种几批，无授权问题，基本都需 OCR | 数据规模 = 个位数到十几批。不可能做统计建模 / ML，但**够做案例**和**够 anchor 用户的隐性经验** |
| 4 | 风险偏好 | 不考虑回报，先看能否做出价值；**最不能接受信誉风险**（客户投入资源后系统不达预期） | **决定性约束** — 排除任何"先拉客户做 design partner"的路径 |
| 5 | 优先突破口 | 提取、制剂；**自己也没有太确切的想法** | 起点不必现在锁定。提取最适合作为第一刀（数据结构化、因果链短），制剂作为延伸 |

### 1.2 用户的元信号（重要）

用户在回答末尾说："**我感觉这事不太明朗**"。

这不是消极信号，而是合理的认知校准——在以上约束下，确实**不应该**现在就锁定一个具体 MVP。"不明朗"本身是一个有效输入：它告诉我们目前的最优策略是 **explore + de-risk**，而不是 commit。

## 2. 战略评估（与 Stage 1 对话同步）

### 2.1 意义

行业意义真实存在，价值锚点分三层：

- **企业层**：codify 老厂长 / 老药工的隐性知识为可审计、可继承的系统资产 — 真正难复制的护城河；
- **行业层**：监管侧（NMPA / 省局）和投资侧都在找"能讲清自己每批药为什么是这个药"的中药企业，差异化筹码明确；
- **科研层**：数据厚度足够后能反哺改良型新药立项 — 从制造跨到 R&D 的少数可行路径之一。

**陷阱**：很多类似项目最终变成"为可视化而可视化"，dashboard 漂亮但没人据此改决策。**真正的意义来自改进建议被采纳并形成闭环**。每一块功能都要回答："这个 dashboard 不存在的话，谁睡不着觉？"

### 2.2 难度

技术难度不是核心。真正的硬骨头：

1. **数据可及性** — 7-10 个分散系统 + 大量纸面 / Excel，归一化本身就是系统性工程；
2. **因果建模** — 长链条、强非线性、长时滞（种植 1-3 年 / 临床数月），相关性挖掘极易得出误导性结论；
3. **合规边界** — RX 临床必须走伦理 + IIT / RWS；OTC 京东评价大数据判定踩到《个保法》《广告法》《医疗广告管理办法》，错一步就是行政处罚级风险；
4. **组织阻力** — 让生产部实时上传参数、QA 公开异常归因、销售接退货数据 — 技术能买，组织变革不能；
5. **标杆从哪来** — 药典只是底线；真正能产生改进建议的"目标值"要从企业历史最佳批反推或同业对标，多数中药厂没有这个数据基础。

工程量级估算：
- 看上去能跑的 demo：6-12 个月、200-400 万人民币级；
- 能进真实生产决策的系统：3-5 年、千万级；
- SaaS 化卖给行业：5-8 年起步。

### 2.3 性价比

| 路线 | 性价比 | 说明 |
|------|--------|------|
| A 单企业内部系统 | 中等偏低 | 除非企业处于扩规模 / 上市 / 融资关口，否则边际改进难以覆盖前期投入 |
| B 开源方法论 + 参考实现（与 huadian D-route 同构） | **高** | 前期投入可控、长尾价值大、能撬动行业资源 |
| C SaaS / 咨询 | 依赖种子客户 | 中药 CMO / 中型品牌方极保守，没有 design partner 起不来；且**与用户"不接受信誉风险"硬冲突** |
| D 组合（用户选） | 关键是先产出"可复用资产"，不要锁定具体路线 | 见下文 §4 |

### 2.4 用户已有资产的契合度

**TraceGuard**：本质是"在多步骤管线中插入 stateless guardian + LLM-as-Judge + 自动修正建议"。把 pipeline 概念换成中药生产工序，guardian 就是"工序节点的标杆数据比对器"。复用点：
- 通用化干净，不绑定具体领域；
- structural + semantic 双层校验 ↔ 中药"硬指标（含量、得率）+ 软判断（性状、气味、汤色）"；
- eval_store + drift detection + suggest 三件套 ≈ 用户要的"沉淀 + 关联挖掘 + 改进建议"雏形。
- **角色定位**：生产侧的运行时底座。

**HuaDian (D-route)**：本质是"用 agent 团队严谨构造可审计知识库的工程框架"。最大资产在方法论层 — 多角色协作、ADR、V1-V11 invariants、audit trail、identity resolver pattern。结构是同构的：
- "史记本纪" → "中药品种工艺规程 + 批历史"；
- "识别司马迁笔下人物" → "识别批记录里的关键工艺偏差 / 异常归因"。
- **角色定位**：知识侧的工程框架。

两者合计已覆盖目标系统骨架的 30-40%。这是不可忽视的存量优势。

## 3. 关键 Reframing — 从"建系统"降维到"建案例"

在用户约束（单人 + AI / 几批数据 / 零信誉风险容忍 / 路线不锁定）下，"建系统"路径在 12 个月内**几乎必败**——不是技术失败，而是组织、客户、数据、合规四面交困都不可解。

但如果把目标换成**"建一份可审计的领域知识工程参考案例"**，所有约束都从障碍变成助力：

| 原约束 | 在"建系统"框架下 | 在"建案例"框架下 |
|--------|---------------|---------------|
| 单人 + AI | 资源严重不足 | 正好是 huadian D-route 最佳工况（shiji-kb 55 小时验证过） |
| 几批 OCR 数据 | 无法训练 / 建模 | **足够** anchor 隐性经验、做方法论 dogfood、跑 traceguard demo |
| 零信誉风险 | 排除一切 design partner 路径 | 不需要客户，自给自足即可产出 |
| 路线不锁定 | 决策瘫痪 | 案例本身就是"将来 A/B/C 都用得上"的 portfolio asset |
| "不太明朗" | 焦虑信号 | 探索期信号 — 案例化是探索的合理形式 |

### 3.1 用户最值钱的资产是什么

不是几批旧批记录。
是**用户作为前制药企业负责人，亲身见过全链条、知道每个工序"哪些参数真的影响下一道工序"、"哪些异常是糟糕的、哪些是无害的"** 这个隐性经验。

这种 tacit knowledge 在制药行业极少被 codify，且会随个人离开行业而流失。**先做的事，应该是把这部分隐性经验形式化为可被 AI 协作迭代的工程资产**——不是先建系统。

## 4. 推荐最小起点（4-8 周 / 单人 + AI / 完全自给自足）

把目标定为：**产出"中成药制造质量决策" 案例 v0.1**，作为 huadian D-route 第二案例候选材料。

### 4.1 范围切割

只做"提取工序"一段。理由：
- 数据相对结构化（投料、设备参数、提取物指标三组数据闭合）；
- 因果链短（不涉及 1-3 年时滞的种植、不涉及合规敏感的临床 / 评价数据）；
- 能复用 traceguard structural + semantic 双层校验现成框架；
- 8 周内可见结果。

### 4.2 产出清单

1. **`docs/cases/tcm-extraction/case-overview.md`** — 案例 framing、范围、与 huadian 框架的同构映射；
2. **`docs/cases/tcm-extraction/data/`** — 用户提供的几批批记录 OCR 后的结构化版本（CSV / JSON）；
3. **`docs/cases/tcm-extraction/invariants.md`** — 用户脑中的隐性经验 codify 为 10-30 条 V-style invariants（"提取液 pH 必须在 X-Y 之间"、"投料药材含量 < Z 时该批应预警"、"提取得率与原料批次的相关阈值"…）；
4. **`docs/cases/tcm-extraction/decision-rules.md`** — "如果出现这种偏差，标杆决策应该是什么"的规则集，作为改进建议生成的 ground truth；
5. **`apps/extraction-guardian/`** — 一个最小 traceguard 配置 + 跑通：输入一份批记录 JSON，输出与历史最佳批比对 + 偏差点 + 改进建议 draft；
6. **`docs/methodology/case-2-tcm-extraction-postmortem.md`** — 方法论复盘："如何从一个老厂长的脑子 + 几批旧数据出发，做出可审计的工艺质量决策案例"。

### 4.3 时间盒

| 周 | 工作内容 | 主要参与者 |
|----|---------|-----------|
| 1-2 | OCR + 结构化批记录 / case-overview 起草 | 用户提供原件 + Claude OCR + 用户校对 |
| 3-4 | 隐性经验访谈 → invariants v0.1 起草 | Claude 主动 propose 提问框架 + 用户回答 + Claude 起草 |
| 5-6 | decision-rules + traceguard 配置接通 | Claude 主导 + 用户校对工艺规则正确性 |
| 7 | demo 端到端贯通 + 复盘文章起草 | Claude 起草 + 用户审稿 |
| 8 | 收档、登记到 huadian D-route Layer 3 候选池 | 架构师（Claude）+ 用户批准 |

### 4.4 为什么这能避开"信誉风险"

- 全程无外部客户参与；
- 数据来自用户本人企业，无第三方利益相关；
- 产出物是文档 + 开源代码，不是承诺给谁的系统；
- 即便最终 demo 不漂亮，也是一份"诚实的探索过程记录"，不是"承诺没兑现"。

**关键事实**：

**成本**：0（.doc 是 Word 原生，可直接提取，无 OCR）

### 10.2 Stage 1 — Pilot OCR（P1 B1 / 16 页 / 2026-05-08 ✅ 通过）

**目标**：验证 Claude vision 端到端 OCR 路径在中药批生产记录（机打模板 + 手写填值 + 400 DPI 扫描）上的可行性。

**实际范围**：P1 B1 提取工序前段 16/100 页（封面 + 领料单 + 称量 + 提取 + 浓缩 + 收膏前段 + 清场卡）

**判定标准 / 实际结果**：
- 关键字段准确率 ≥ 95% — ✅ 用户人工抽查 5 页（p.4, 7, 8, 9, 12）全部正确
- 数值字段（kg / 温度 / 压力 / 时间）≥ 98%
- 物料代码 / 设备编号 ≥ 95%
- 签名识别 ~70%（潦草处占位 "(签名)"）

### 10.2.1 F-001 Fundamental Finding（Stage 1 用户答疑后浮现 / 2026-05-08 / pilot-status-v0.2.md）

这意味着 v0.1 解读"料液比 6.000 / 5.000 倍精确吻合规程 = 严格执行"是错的，**正确解读是"反向计算填表"**。

由此引发的命题升级：

- **v0.1 命题**："规程外的经验才是 quality-decisive"
- **v0.2 命题**：**"中药批生产记录是分层制品 (layered artifact)。表层是 GMP 合规叙事 / 中层是实际工序执行（大部分未写）/ 深层是身体化经验。三层之间存在结构性代差。任何想用 AI 改进中药制造的努力，必须显式区分这三层。"**

战略含义：
- 本案例的开源/方法论价值显著上升 — 它提出了一个"学界、监管、产业都还没正面回答的问题"
- D-route Layer 2 方法论文章可孵化新选题："如果一个行业的合规记录不是 ground truth，AI 改进的入口在哪里？"
- 用户最初提到的"通过视觉/视频/IoT 多模态采集 enrich 缺失部分"的思路获得**理论升级**：不是锦上添花，而是中/深层 ground truth 的**唯一**采集路径

详细论述见 `cases/tcm-extraction/data/pilot-WK-B1/pilot-status-v0.2.md` §2 + §3。
