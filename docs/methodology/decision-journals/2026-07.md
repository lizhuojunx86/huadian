## 2. 关键元决策（按时间）

### 2.1 真相表的设立 — 给 ⭐ 叙事配一个"去乐观偏差对照"（6/7）

**上下文**：6/7 进度评估发现一个治理层问题：STATUS.md 用大量 ⭐ 和"领先 6-8 周"渲染容易产出的维度（文档 / sprint 计数），系统性盖住了对外可验证里程碑的真实状态（当时 push 停在 5/15、发表 0、外审 0、L1 冻结 5 周）。**这不是欺骗——限定词都在——而是渲染层与判据层未分离。**

**决策**：新增 STATUS §1.2「对外可验证判据真相表」，11 项、二元 done/not-done、与 ⭐ 叙事物理分离、每次 sync 强制更新。同时 L1 状态色 🟢→🟡、L4 🟢→⚪（名实对齐）。

**元决策反思**：自我评估机制本身会被"容易度量的东西"俘获——这与 case-2 研究的 F-002（话语偏向）在结构上是同一件事，只是发生在**我们自己身上**。解法也同构：分通道（内部产能叙事 vs 外部可验证判据），不许汇总成单一"进展良好"总分。**研究别人的度量失真时，先给自己装同款检测器。**

### 2.2 兑现模式切换 + P0 脱敏险情（6/10 → 6/13）

**上下文**：6/10 十-agent 方向评估结论：D-route 方向保留，但价值兑现全部卡在"公开输出 + 外部接触"；价值排序（case-2 文章 > 过程标本 > 方法论 > 框架代码）与 L1 优先级倒挂。用户 ACK"方向保留、兑现模式从闭门精修切换为公开输出"。

**随即暴露的险情**：发布前扫描发现敏感面远超预估——case-2 数据树含企业名、供应商、人员真名、批号，**乃至国药准字批文号（可经药监局反查企业）**，且已随 5 月的 push 进入公开 git 历史。处置：立即转 private 止血 → 37 文件约 2300 处脱敏 + 6 项重命名（数值计算输入全保留，只切断关联链）→ filter-repo 历史重写 → 逐文件核验后双仓重新 public（6/13）。

**元决策反思（两条）**：
1. **"公开"是模式切换，不是在原工作流上加一个发布动作。** 闭门期积累的所有资产都默认"只有我们看"，公开的第一步必然是一次全量的敏感面审计。此教训已固化：任何新产出如果未来可能公开，脱敏检查应前置为写入时纪律，而非发布前补课。
2. 险情的处置顺序（止血 → 映射确认 → 批量替换 → 历史重写 → 核验 → 重新公开）本身是一个可复用的 incident playbook——它和红线纪律一样，价值在于**下次不用重新想**。

### 2.3 L1 解冻的方式：先证伪自己的整改方案（6/20 / ADR-039）

**上下文**：L1 代码自 5/3 冻结，止损线 S5 已触发。原整改方案假设"领域无关比例偏低 = 有 huadian 专有逻辑泄漏进 core，应下沉到 examples"。

**调研结果证伪了方案前提**：core 实测 0 领域字样，64.4% 这个比例偏低是"单一 example 过重"的**度量假象**，不是工程缺陷——无可下沉项。真正的杠杆是 ① import-root 改名（`framework.*`→`kb_forge.*`）和 ② 引入第 2 真实 domain。用户选 ①+③（判据重定义），当日执行：目录改名 + 73 imports / 51 文件 + pyproject + 13 活文档，pytest 93 passed / 0 回归。判据从失真的"≥70% LOC"重定义为"core+tests 0 领域字样 ✅ + ≥2 domain 隔离（1/2）"。

**元决策反思**：这是 §2.1 的续篇——**判据本身也会失真**。"≥70% LOC"在单 example 仓里天然不可达，按它整改会做无用功（搬不存在的泄漏）。正确顺序是：整改前先验证整改方案的前提；前提错了，改判据而不是硬凑数字。判据重定义不是粉饰，检验标准是"剩余 gap 是否显式保留"（第 2 domain 1/2 就是保留的 gap）。

### 2.4 分发的第一批真实读数 + 第一个外部信号（6/21-6/23）

**读数（6/21 分发战术备忘）**：发布 2 周，唯一接近"质量信号"的是第 4 篇拿去**回答已有问题**得到的 5 个收藏（回答 70 阅读 vs 专栏 40）；其余（首 star、23 访客、328 clones）统计上全是噪声。机制核实：知乎专栏推荐权重仅为回答的 1/5~1/10——"回答式分发"是渠道机制决定的正确方向，但 n=1 不构成证明。**纪律**：数据每 2-4 周看一次、策略每季度调一次、不因样本量=4 的数据改选题/人设/风格。

**外部信号（6/22）**：shiji-kb 作者 baojie（2112★ / 30+ 年资历）在 Discussion #114 实质回复——"多角色 agent team 思路是对的、往下能走得通"，并自述兴趣已从史记转向 harness 框架（与 kb-forge 同赛道）。真相表「近邻交流」提前目标锚（2026-10）约 4 个月达成。

**元决策反思**：分发期最大的风险不是"没人看"，而是**用 2 天粒度检视一个 2-5 年的慢变量**，然后把噪声当信号去"优化"。本月立的"降采样频率"纪律，和 §2.1 真相表、§2.3 判据重定义一样，都是同一主题的变体：**度量什么、多久度量一次，本身是要被设计和防御的决策。**

### 2.5 静默期（6/23 → 7/11）— 与 5 月"沉淀期"的性质区分

**事实**：6/29 落了两个 commit（traceguard side-write bridge + SHA pin，L1「代码推进」判据延续）之后，至本篇起草日约 2 周无产出，本日记也因此迟到。

**与 5 月沉淀期的区别（必须诚实区分）**：5/15-5/31 的沉淀期是**主动设计的节奏工具**（有明确的进入/退出条件，期间只走轻量并行项）；6/23 后的静默是**无设计的停摆**——没有决策说"现在休整"，只是没有动。二者不可混称。

**根因判读**：剩余高价值动作全部是**用户侧真人动作**（联系外审人 / 注册 SSRN / 论坛发帖），而非架构师可代笔的写作。写作产能过剩 + 外部动作停滞 = 停摆的真实结构。6/20 外审 SOP 已把动作拆小，但"拆小"不等于"发生"。

**元决策反思**：瓶颈已经第三次转移——5 月是**写作产能**（架构师起草解决），6 月初是**审稿消化**（错峰解决），现在是**外部真人动作**。前两类瓶颈都能靠架构师多做来缓解；第三类恰恰相反，架构师做得越多（再写一篇 SOP、再优化一版话术），越容易制造"在推进"的假象。正确动作是把每个外部动作**包装到只剩点击**（成品信件 / 成品发帖包 / 分步指南），然后停笔等执行。7/11 本次会话即按此执行。

## 3. 元反思 — 协作模式进入第三形态

按 CLAUDE.md §6.2：用户写作偏弱，架构师主动 propose + 起草，用户审稿 + 决策。

- **第一形态（5 月）**：架构师起草长文，用户审稿。瓶颈=写作产能。
- **第二形态（6 月初）**：架构师爆发式收割，用户错峰消化。瓶颈=审稿带宽。
- **第三形态（本月显形）**：高价值动作移到外部世界（外审 / 预印 / 论坛 / 社区），这些动作**只有用户能做**。架构师的正确角色从"多写"变为"把外部动作的摩擦降到最低 + 把信号回收成下一步决策"。

**一条对称的诚实提醒**：第三形态下，架构师侧的过度产出（更多 SOP、更多计划文档）会成为**新的拖延形式**。本月的自查标准应从"架构师产出了什么"改为"用户完成了几个外部动作、回收了什么信号"。
