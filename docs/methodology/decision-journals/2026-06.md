## 2. 关键元决策（按时间）

### 2.1 "沉淀期"作为刻意节奏工具的回看验证（5/15→5/31）

**上下文**：5 月日记 §2.5 做了一个"领先 vs 节制"的张力决策——09 v0.2 全本完成、路线图领先 ~6-8 周后，**主动选择节制**，给 GO-J-ε 后留 1-1.5 个月沉淀期，而非立刻冲 v0.3 完整初稿。

**回看判断**：这个决策在本月初看是对的。沉淀期内只走了一个轻量、非阻塞的并行项（GO-M-ζ schema v0.4 outline / D5 路径），没有为"赶进度"而堆叠浅层产出。沉淀期结束后第一步（GO-N-η）是把沉淀期内起草的 outline 收成完整 lock——**节奏自然、无返工**。

**元决策反思**：当一个项目"领先于自己的路线图"时，默认动作不应是"用领先量再加速"，而应是"把领先转化为质量缓冲"。沉淀期把"领先 ~6-8 周"变成了"v0.3 启动前 schema 锚点已稳"的实际收益，而不是又一批待返工的草稿。

**对未来的启示**：沉淀期是可复用的节奏工具，不是"停工"。Layer 2 文章 v0.3 → v0.4 → v1.0 之间也应有类似的"审稿沉淀窗"。（注：本启示在 6/1 爆发后获得新意义，见 §2.5。）

### 2.2 schema v0.4 完整 lock — versioning 工作流二次走通（GO-N-η / 6/1）

**关键判断**：case-2 schema 的演进采用"records-only patch 先行（v0.3.1~v0.3.4）→ 触发条件累积 → 独立 ADR + 正式 lock"的工作流。这一工作流在 v0.2→v0.3（ADR-034）首次走通，本月 v0.3→v0.4（ADR-035 + GO-N-η）**第二次走通**。

**框架抽象意义**：这不是 case-only 的完成度工作。它实证了 06 文章 §8（ADR Template Comparison Pattern）里"schema versioning + 小步迭代"的可复用性——一个领域无关的工程模式有了第 2 个活体数据点（继 case-1 的 V1-V11 / migration 链之后）。

**元决策反思**：把"已在数据里 candidate 落地的字段"和"已正式 lock 进 schema 文档的字段"显式分层（records-only patch vs schema lock），让案例既能快速吸收新发现、又不丢 schema 的严谨性。这层"双速"机制本身是值得抽象的方法论。

### 2.3 红线纪律的一次干净实证 + 一个 doc-sync 盲点（GO-N-η batch-record.json _meta 修改）

**事件**：v0.4 完整 lock 需要改 batch-record.json `_meta`（schema_version + history）。这命中 CLAUDE.md §5 红线 #2（覆盖已有文件），ADR-035 §6.3 也预先标了"需单独 ACK"。执行前走了确认格式（影响范围 / 后果 / 回滚），用户 ACK 后才动手。

**元决策反思**：红线流程在"改动其实很小"（2 字段）时也照走，没有因为"改动小"就跳过确认。这是纪律的价值——纪律的强度不应随单次改动的大小浮动，否则边界会被逐步侵蚀。

**一个 doc-sync 盲点（6/2 复盘发现）**：`_meta` 实际已升 0.4（JSON + `_schema_upgrade_history` 已记），但 `extraction-schema-v0.4.md` 正文仍留 4 处"⏳ 待 ACK"措辞未回填。红线"改前确认"执行得干净，"改后把所有引用点同步到新事实"却滞后了。**启示**：红线的盲点往往不在改动本身，而在改动后**散落各处的旧表述未被一并更新**——doc-sync 应作为红线操作的收尾步纳入清单（待同批修订）。

**对未来的启示**：08/09 文章 v0.4 审稿期间若需回改 batch-record.json 或 schema lock 文档，同样按红线 #2 处理，并把"同步所有引用点"列入收尾。

### 2.4 本月写作节奏决策 — 08+09 v0.3 完整初稿当日达成（实况校正草稿预测）

**v0.1 草稿原预判（6/1 GO-N-η 当刻）**：08 v0.3 是 4-6 周工作、分批章节展开、09 留到 2026-07，双文章保持"错峰"。

**6/1 实际**：当天 GO-O-η~GO-T-η 把 08 v0.3 精炼版（4015 字）+ 09 v0.3 精炼版（4054 字）一并收完——两篇配对**同步**而非错峰，预测的 4-6 周压缩成 1 天。

**为什么压缩成立（不是赶工）**：v0.3 不是"从零写"，而是把 v0.2 全本 + GO-N-η 焊稳的 schema 锚点（F-001 v0.4 / F-004 / F-002 vs F-004 对比表）**收割**成精炼形态。锚点稳 → 写作降为"精简 + 补文献综述 + 存快照"的收尾动作 → 单日可控。

**压缩中守住的纪律**：每阶段 research-first（引用 web 核实，如临床 Studdert 2005 / 金融 Burgstahler&Dichev 1997）、存全量快照后再精简（无损 / archive/）、严格落在 4000-6000 字目标。**压缩 ≠ 潦草**。

**与 09 的关系（原计划作废）**：草稿里"09 留 2026-07 错峰"已不成立——双文章 v0.3 当日同步达成；下一个真实配对节点是 v0.4 审稿（ETA 2026-09）。

### 2.5 「节制 vs 6/1 爆发」的张力（本月最值得记的元决策 / v1.0 新增）

**表面矛盾**：本篇 §2.1 + §3 颂扬沉淀期与"节制即写作能力"，6/1 却单日连走 7 阶段。看上去自相矛盾。

**我的判读（不矛盾 / 时间分离 · 功能互补）**：节制发生在沉淀期（5/15→5/31）——克制住"再加速堆草稿"的冲动；爆发发生在 6/1——把沉淀期焊稳的锚点一次性收割成精炼形态。前者防"浅层堆量"，后者做"深度固化"，二者不在同一维度对立。**沉淀期恰恰是 6/1 能干净爆发的前提**（§2.1 "把领先转成质量缓冲"在此兑现）。

**但要诚实记一条成本**：08+09 两篇 v0.3 同日落地，意味着**审稿负荷被前置且叠加**——用户的审稿台现在同时摆着两篇完整 v0.3，而非错峰各审一篇。这是"配对同步爆发"的代价。

**对未来的启示**：v0.3→v0.4 审稿阶段应主动**错峰**（先 08 后 09，或反之），把同步爆发省下的时间还给审稿，避免审稿成为下一个瓶颈。

**元决策反思**：节奏工具（节制 / 爆发 / 错峰）应按"当前瓶颈在哪"动态换挡。写作产能不再是瓶颈时，瓶颈转移到审稿——节奏调度的对象也要随之从"产出"转向"消化"。

## 3. 元反思 — 协作模式持续 + "节制"与"爆发"是同一种产能调度能力

按 CLAUDE.md §6.2 "承认能力短板"原则：用户写作偏弱 / 架构师主动 propose 框架 + 起草初稿 / 用户角色为"审稿 + 决策 + ground truth 提供"。

**本月实证（6/1 全程）**：

- GO-N-η 的 schema-v0.4 完整 lock 文档、08+09 两篇 v0.3 完整初稿（精炼 + 文献 + 快照）全部由架构师起草/精简；用户角色 = 红线 ACK（batch-record.json 修改授权）+ 推进顺序决策（先日记、再 08、再 09）。
- 本篇月度日记同样由架构师起草、待用户审稿——延续 5 月日记 §3 的协作闭环（第 2 次）。

**本月新增反思**：5 月日记把"写作能力短板"主要理解为"起草由架构师承担"。本月在两端各补一条：（1）§2.1 的"节制不写"是一种写作能力；（2）§2.5 的"6/1 爆发"说明，"节制"与"爆发"其实是**同一种能力的两面**——按瓶颈状态调度产能。真正稀缺的不是"能多写"或"能少写"，而是"知道此刻该写还是该停、该同步还是该错峰"的判断力。这条判断力对用户同样适用：审稿环节的价值在 v0.3/v0.4 阶段显著加重。

**对未来的启示**：
1. 协作模式（架构师起草 + 用户审稿）在 2026-10 双文章 v1.0 发布前持续保持。
2. v0.4 审稿阶段用户审稿环节显著加重，且按 §2.5 主动错峰，避免两篇同审。
3. 本篇日记按"v0.1 草稿（6/1）→ 用户审稿 → v1.0 定版（6/2）"流程走，给审稿留了跨日窗口（未在起草同日强行定版）。
