## 2. 关键元决策（按时间）

### 2.1 "沉淀期"作为刻意节奏工具的回看验证（5/15→5/31）

**上下文**：5 月日记 §2.5 做了一个"领先 vs 节制"的张力决策——09 v0.2 全本完成、路线图领先 ~6-8 周后，**主动选择节制**，给 GO-J-ε 后留 1-1.5 个月沉淀期，而非立刻冲 v0.3 完整初稿。

**回看判断**：这个决策在本月初看是对的。沉淀期内只走了一个轻量、非阻塞的并行项（GO-M-ζ schema v0.4 outline / D5 路径），没有为"赶进度"而堆叠浅层产出。沉淀期结束后第一步（GO-N-η）是把沉淀期内起草的 outline 收成完整 lock——**节奏自然、无返工**。

**元决策反思**：当一个项目"领先于自己的路线图"时，默认动作不应是"用领先量再加速"，而应是"把领先转化为质量缓冲"。沉淀期把"领先 ~6-8 周"变成了"v0.3 启动前 schema 锚点已稳"的实际收益，而不是又一批待返工的草稿。

**对未来的启示**：沉淀期是可复用的节奏工具，不是"停工"。Layer 2 文章 v0.3 → v0.4 → v1.0 之间也应有类似的"审稿沉淀窗"。

### 2.2 schema v0.4 完整 lock — versioning 工作流二次走通（GO-N-η / 6/1）

**关键判断**：case-2 schema 的演进采用"records-only patch 先行（v0.3.1~v0.3.4）→ 触发条件累积 → 独立 ADR + 正式 lock"的工作流。这一工作流在 v0.2→v0.3（ADR-034）首次走通，本月 v0.3→v0.4（ADR-035 + GO-N-η）**第二次走通**。

**框架抽象意义**：这不是 case-only 的完成度工作。它实证了 06 文章 §8（ADR Template Comparison Pattern）里"schema versioning + 小步迭代"的可复用性——一个领域无关的工程模式有了第 2 个活体数据点（继 case-1 的 V1-V11 / migration 链之后）。

**元决策反思**：把"已在数据里 candidate 落地的字段"和"已正式 lock 进 schema 文档的字段"显式分层（records-only patch vs schema lock），让案例既能快速吸收新发现、又不丢 schema 的严谨性。这层"双速"机制本身是值得抽象的方法论。

### 2.3 红线纪律的一次干净实证（GO-N-η batch-record.json _meta 修改）

**事件**：v0.4 完整 lock 需要改 batch-record.json `_meta`（schema_version + history）。这命中 CLAUDE.md §5 红线 #2（覆盖已有文件），ADR-035 §6.3 也预先标了"需单独 ACK"。执行前走了确认格式（影响范围 / 后果 / 回滚），用户 ACK 后才动手。

**元决策反思**：红线流程在"改动其实很小"（2 字段）时也照走，没有因为"改动小"就跳过确认。这是纪律的价值——纪律的强度不应随单次改动的大小浮动，否则边界会被逐步侵蚀。

**对未来的启示**：08/09 文章 v0.3 写作期间若需回改 batch-record.json 或 schema lock 文档，同样按红线 #2 处理。

## 3. 元反思 — 协作模式持续 + "节制"也是一种写作能力

按 CLAUDE.md §6.2 "承认能力短板"原则：用户写作偏弱 / 架构师主动 propose 框架 + 起草初稿 / 用户角色为"审稿 + 决策 + ground truth 提供"。

**本月初实证（截至 6/1）**：

- GO-N-η 的 schema-v0.4 完整 lock 文档由架构师起草；用户角色 = 红线 ACK（batch-record.json 修改授权）+ 推进顺序决策（先日记、再 08 v0.3）。
- 本篇月度日记同样由架构师起草、待用户审稿——延续 5 月日记 §3 的协作闭环。

**本月新增反思**：5 月日记把"写作能力短板"主要理解为"起草由架构师承担"。本月沉淀期的回看（§2.1）补了一条：**"节制不写"也是一种需要刻意行使的写作能力**。在项目领先于路线图时，克制住"再加速"的冲动、把领先转成质量缓冲，比多产出一批草稿更难、也更有价值。这条对用户同样适用——审稿环节的价值在 v0.3/v0.4 阶段会显著加重，不必急于在 v0.2 之后立刻冲刺。

**对未来的启示**：
1. 协作模式（架构师起草 + 用户审稿）在 2026-10 双文章 v1.0 发布前持续保持。
2. v0.3 完整初稿期间，用户审稿环节应显著加重（v0.3 是学界审稿前的最后形态）。
3. 本篇日记按"v0.1 草稿 → 用户审稿 → 定版"流程走（不在起草同日强行定版，给审稿留窗口）。
