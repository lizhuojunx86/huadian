# 04 工作流与进度机制

> 本文件解决用户明确提出的顾虑：
> "我对电脑编程不太熟悉，每个环节应考虑我如何高效地接续上之前的工作。"
>
> 核心设计目标：
> 1. **任何时候打开项目**，5 分钟内能看清当前在哪、下一步做什么
> 2. **任何决策变更**都留有书面痕迹，能说明原因
> 3. **跨会话接续友好**，不需要重新解释背景
> 4. **用户无需写代码即可参与决策审批**

---

## 一、四件套（项目的"神经系统"）

```
docs/
├── STATUS.md                       # ① 活状态板（每次会话更新）
├── CHANGELOG.md                    # ② 时间线（追加，不修改已有条目）
├── decisions/                      # ③ 架构决策记录（ADR）
│   ├── ADR-000-index.md
│   ├── ADR-001-single-pg.md
│   ├── ADR-002-event-account-split.md
│   └── ...
└── tasks/                          # ④ 任务卡
    ├── T-000-index.md
    ├── T-001-monorepo-skeleton.md
    └── ...
```

### ① STATUS.md — 活状态板
**作用**：项目的"现在时刻"。每次会话开始、结束必读。
**更新频率**：每次有进展就更新。不是日志，是快照。
**内容结构**：
- 当前阶段（Phase / Sprint）
- 进行中的任务（`in_progress` 任务的 ID + 负责角色）
- 已完成（本 Sprint）
- 下一步候选（`pending` 任务的建议执行顺序）
- 阻塞项（等待用户决策 / 外部依赖）
- 最近的 3-5 个 ADR 链接

### ② CHANGELOG.md — 时间线
**作用**：项目演进史。只追加，不修改已有条目。
**更新频率**：每次会话结束 / 每个任务完成 / 每次决策变更。
**每条条目格式**：
```
## YYYY-MM-DD HH:MM
### [类型] 标题
- 角色：{role}
- 任务：T-NNN（链接到任务卡）
- 变更：具体做了什么
- 影响：影响哪些文件 / 哪些角色 / 哪些数据
- 原因：为什么
```
类型枚举：`feat` / `fix` / `refactor` / `docs` / `chore` / `decision` / `constitution` / `schema`

### ③ ADR（Architecture Decision Record）
**作用**：记录"为什么这样设计"，让新成员 / 未来的自己能理解。
**更新规则**：
- 任何非琐碎的设计选择都需要 ADR
- 一旦提交，**不修改**原内容；如果决策改变，新增 ADR 并标注 `supersedes: ADR-NNN`
- 文件名：`ADR-NNN-kebab-title.md`
- 模板见下文

### ④ 任务卡（Task Card）
**作用**：每个具体执行任务的"工单"。
**生命周期**：`pending → in_progress → review → done`
**模板见下文**

## 二、ADR 模板

文件：`docs/decisions/ADR-NNN-title.md`

```markdown
# ADR-NNN: {标题}

- **状态**：proposed / accepted / deprecated / superseded
- **日期**：YYYY-MM-DD
- **提议人**：{role}
- **决策人**：{role/用户}
- **影响范围**：{模块/表/流程}
- **Supersedes**：ADR-XXX（如有）
- **Superseded by**：ADR-YYY（如有）

## 背景
{为什么需要这个决策}

## 选项
### 选项 A：...
优点 / 缺点
### 选项 B：...
优点 / 缺点
### 选项 C：...
优点 / 缺点

## 决策
我们选择 {A/B/C}，理由是...

## 影响
- 正面：...
- 负面：...
- 迁移成本：...

## 回滚方案
{如何撤销}

## 相关链接
- 任务卡：T-NNN
- 相关 ADR：ADR-XXX
- 代码 PR：#NNN
```

## 三、任务卡模板

文件：`docs/tasks/T-NNN-title.md`

```markdown
# T-NNN: {任务标题}

- **状态**：pending / in_progress / review / done / blocked
- **主导角色**：{role}
- **协作角色**：{roles}
- **所属 Phase**：Phase 0 / 1 / 2 / 3 / 4
- **依赖任务**：T-XXX, T-YYY（完成才能开始本任务）
- **预估工时**：{N 人日}
- **创建日期**：YYYY-MM-DD
- **开始日期**：
- **完成日期**：

## 目标（Why）
{为什么要做这个任务，达成什么业务价值}

## 范围（What）
### 包含
- ...
### 不包含（重要！防止 scope creep）

## 交付物（Deliverables）
- [ ] 代码：{路径}
- [ ] 文档：{路径}
- [ ] 测试：{路径}

## 完成定义（DoD）
- [ ] 代码合入 main
- [ ] 相关测试通过
- [ ] 文档更新
- [ ] STATUS.md 更新
```

## 四、标准任务流程

```
 ① Kickoff
    ├─ 主导角色读 STATUS + 任务卡 + 角色文档
    ├─ 生成 TodoList
    └─ 用户确认是否开始 ← 【用户参与点】

 ② Plan
    ├─ 拆分可测量的子任务
    ├─ 标注每个子任务的验收标准
    └─ 识别协作需求（找哪个角色要什么）

 ③ Execute
    ├─ 按 TodoList 执行
    ├─ 每完成一个子任务 commit（conventional format）
    └─ 触发危险操作时先确认 ← 【用户参与点】

 ④ Review
    ├─ QA 审查
    ├─ 其他协作角色签收
    └─ 更新文档

 ⑤ Close
    ├─ 更新 STATUS.md
    ├─ 追加 CHANGELOG.md
    ├─ 任务卡状态 → done
    └─ 标注 handoff_to（如需）
```

## 五、用户参与点（给用户的介入锚）

作为用户（不熟编程），以下是你必须介入的关键节点：

| 节点 | 何时 | 做什么 |
|------|------|-------|
| Kickoff 确认 | 任务启动前 | 阅读任务卡，确认目标与范围 |
| 危险操作确认 | CLAUDE.md 红线触发时 | 根据确认格式回答"是/否" |
| ADR 决策 | 新增 ADR 且标注"需用户决策" | 选择选项 |
| Sprint 回顾 | 每周末 | 看 STATUS.md，判断是否按预期推进 |
| PRD 审批 | PM 提交新 PRD | 阅读 + 批准/调整 |
| 视觉评审 | 设计师出原型 | 确认风格、交互、信息层级 |

**不需要介入的**：
- 具体代码实现
- 测试用例编写
- 日常 commit
- 小型重构

## 六、会话接续模板（直接可用）

### 6.1 恢复工作标准模板
```
继续华典智谱项目。请按以下步骤启动：

1. Read CLAUDE.md（项目入口）
2. Read docs/STATUS.md（当前状态）
3. Read docs/CHANGELOG.md 最近 5 条（最近变更）
4. 告诉我：
   - 当前在哪个 Phase / Sprint
   - 最近完成了什么
   - 当前进行中的任务（如有）
   - 建议接下来推进的任务 ID
5. 等我确认推进的任务 ID 后，再读对应任务卡和角色文档，然后生成 TodoList。
```

### 6.2 切换角色模板
```
本次会话需要切换角色。现在你担任 {新角色}：
1. Read .claude/agents/{新角色}.md
2. Read 任务 T-NNN
3. 说明你作为新角色对本任务的理解
4. 列出你的 TodoList
等我确认后开始。
```

### 6.3 决策征询模板
```
我需要你以首席架构师身份对以下情况做决策：
{情况描述}
请：
1. 列出可能的选项（2-4 个）
2. 给出各选项的优劣与成本
3. 给出你的推荐 + 理由
4. 如果我同意，立即生成对应 ADR
```

### 6.4 Dailly Standup（每日/每次开工速览）
```
给我一个 5 行以内的项目快报：
- 当前 Phase / Sprint
- 进行中任务
- 最近 1 天的进展
- 阻塞项
- 今天建议做什么
```

## 七、版本管理与分支策略

### 分支规则
- `main` — 稳定分支，只接受已评审的 PR
- `develop` — 集成分支（Phase 1 起启用）
- `feat/T-NNN-title` — 任务分支
- `fix/...` / `chore/...` / `docs/...` — 其他类型

### Commit Message（Conventional Commits）
```
feat(pipeline): 增加 NER 抽取步骤 [T-005]
fix(api): person query 返回空导致 500 [T-012]
docs: 更新架构文档 §3.2
refactor(db): 将 events 拆分为 events + event_accounts [T-008, ADR-002]
chore: 升级依赖
```

### PR 流程
1. 任务分支开发 → 提 PR 到 `main`（MVP 阶段）/ `develop`（后期）
2. PR 描述必须包含：任务 ID、交付物清单、测试说明、相关 ADR
3. 用户/架构师批准 → 合并
4. 合并后关闭任务卡

## 八、用户参与的"最小参与模式"

如果你某段时间特别忙，可以采用：

```
我最近没时间详细过每个任务，请按以下方式协作：
1. 所有危险操作（红线）一律暂停等我
2. 所有 ADR 决策暂停等我
3. 其他任务按任务卡推进
4. 每天给我一份 Daily Standup（用 6.4 模板）
5. 每周五给我 weekly summary（过去 7 天做了什么、下周建议）
```

这样项目能自驱运行，你只在关键点介入。

## 九、失败模式与防护

| 失败模式 | 防护措施 |
|---------|---------|
| 任务 scope creep | 任务卡的"不包含"清单强制填写 |
| 决策来回翻 | 新 ADR 必须标注 `supersedes`，不可默默变更 |
| 多个角色 concurrent 改同一文件 | 任务卡的"影响文件"字段 + git branch 隔离 |
| 用户遗忘项目进度 | STATUS.md + Weekly summary |
| 某角色过载 | 任务卡的"协作角色"字段提示可分拆 |
| 文档陈旧 | 每次任务完成必须更新 STATUS + CHANGELOG |

## 十、本机制的自检

每月一次，由架构师运行：
- [ ] STATUS.md 是否反映真实状态（比对 git log）
- [ ] CHANGELOG.md 是否有时间跳空
- [ ] ADR 是否有"未决"超过 14 天的
- [ ] 任务卡 `in_progress` 超过 2 周的需重新评估
- [ ] 角色越位次数（从 PR / commit 分析）

不合格项记入 `docs/retros/YYYY-MM.md`。
