# 给 AI agent 团队设计"硬中断"：5 类 Stop Rule 与 4 个反模式

让 AI agent "跑得快"已经不是难题了。难的是另一件事：**让它知道什么时候必须停下来叫人**。

我在做一个开源项目「华典智谱」（[github.com/lizhuojunx86/huadian](https://github.com/lizhuojunx86/huadian)）：用一组 AI agent 角色（架构师、管线工程师、古籍专家等）从《史记》原文构建一个可审计的人物知识库，同时把过程中沉淀的工程模式抽象成领域无关的框架。跑了十几个迭代之后，最有价值的发现之一不是任何抽取技巧，而是一份"中止条件清单"。两个真实场景：

**场景一**：一次数据回填操作，dry-run 报告预测会写入 179 行，实际 apply 只写入了 175 行。差 4 行，agent 完全有能力给自己找个理由——"大概是去重了吧"——然后继续跑。但我们预先写了一条规则：写入数与预测数不一致即停。Agent 停下来上报，人工诊断后确认是幂等机制拦截了上一轮迭代已写入的旧数据——这次是预期行为。但如果不是呢？如果是 4 行静默丢失呢？没有这条规则，我们永远不会知道这个问题存在过。

**场景二**：一次 LLM 抽取任务，我们给单阶段成本设了 $1.80 的硬上限，实际跑完 $0.79，远没触发。这条规则看起来"白写了"——但它换来的东西很实在：我可以放心让 agent 无人值守地跑完整个阶段，因为我知道最坏情况下损失有界。**Stop Rule 的价值不在触发次数，在于它让"放手"变得安全。**

这篇文章把我们沉淀的 5 类 Stop Rule 模板和 4 个反模式整理出来。它们来自一个古籍知识库项目，但设计是领域无关的——每节末尾会给跨领域的适配提示。

## 什么是 Stop Rule

Stop Rule 是**预先写进任务简报、可量化的中止条件**：任何时点触发，agent 立即暂停当前工作、保留现场、升级人类决策。它不是 code review 那种"反馈建议"，是硬中断。

四条设计原则：

1. **可量化**——必须有具体阈值（"成本 > $X""新实体 > Y 个"），不允许"差不多"
2. **预先声明**——任务开始前显式列出，执行中不允许"补"
3. **覆盖五类风险**——即下文的 5 类
4. **附带处置路径**——每条规则标注触发后该做什么（修因重试 / 回滚 / 接受并记债 / 中止任务）

## 类 1：成本类

```markdown
- 本阶段 LLM cost > $N → Stop
  - 触发时：暂停调用，评估根因（prompt 过长 / batching 失效 / 模型选型）
  - 路径：A. 调参继续 / B. 换更便宜的模型 / C. 中止任务
```

**实证**：摄入《史记·高祖本纪》那轮，上限 $1.80，实际 $0.79；摄入《项羽本纪》那轮，上限 $0.80，实际 $0.60。两次都没触发——但正因为有上限，这两轮都是放心交给 agent 整段执行的。

**跨领域提示**：阈值要按你的文本形态校准。古籍 NER 的 prompt 不长，$1-2 一轮够用；法律合同条款动辄超长上下文，可能要 $10+；医疗领域则可能由合规要求反过来决定预算。重点不是数字本身，而是**写下一个具体数字**。

## 类 2：数据正确性类

```markdown
- 任一不变量（invariant）测试在新数据上 ≠ 0 → 立即 Stop
  - 触发时：不允许"先入库再修"
  - 路径：A. 修因后重试 / B. 回滚 + 中止

- 黄金集（golden set）任一已有章节回归 → Stop
  - 触发时：不允许"用已有数据的退步换新数据上线"
```

不变量是指知识库里**绝不能为假的断言**——比如"每条抽取结果必须能追溯到原文位置"这类性质，做成自动化测试，每次数据变更后全量跑。我们项目维护了 22 个这样的测试，要求永远全绿。

**实证**：多轮摄入中这两条规则都未触发（不变量保持 0 违例、黄金集无回归）。这恰恰是它们在工作的证据：上游 prompt 和合并逻辑的每次改动，都是在"改坏了立刻会被拦下"的约束下进行的。

**跨领域提示**：换领域时不变量要重新定义——合同知识库可能是"当事方一致性""条款引用完整性"，医疗则是临床指南或 HIPAA 合规相关断言。这是五类规则里**最不该省**的一类：成本超了能补，错误数据入库后污染下游，回滚成本极高。

## 类 3：输出量类

```markdown
- 本阶段新增实体 > N 个 → Stop，进入批量审查
  - 触发时：警告而非立即中止；判断是模型过度抽取还是数据本身就丰富
  - 路径：A. 接受继续 / B. 调整抽取 prompt / C. 拆成子批次处理
```

输出量异常是 LLM 工作流里最隐蔽的失败：多抽出 300 个"实体"在仪表盘上看起来像高产出，实际可能是噪声洪水。**Agent 的产出量和产出质量没有天然相关性**，必须用量的异常做质的预警。

**实证**：高祖本纪那轮设了"新增人物 > 120 即停"，实际新增 85，未触发。另一条反向的：同名人物跨国别误合并（《史记》里同名的王侯非常多）是我们的高频错误源，于是设了"自动拦截治理率 < 70% 即停"——那轮实测 100%，规则没拦住任务，反而成了庆祝指标。Stop Rule 的阈值同时也是健康度的标尺。

**跨领域提示**：数量阈值必须校准到你领域"一轮正常输出"的量级，拿前两三轮的实际数据定，别拍脑袋。

## 类 4：治理类

```markdown
- 领域专家对自动合并候选的 reject 率 > N% → Stop
  - 触发时：说明自动决策准确率不够，不该继续喂数据
  - 路径：A. 加强自动拦截规则 / B. 调抽取 prompt / C. 接受 + 写入复盘

- 人工 review 工时 > N 小时 → Stop
  - 触发时：review 负担过重，先改工具再继续
```

这一类规则监控的不是数据，是**人机协作本身**的健康度。

**实证**：一次摄入后，古籍专家角色审核自动合并候选时拒绝了 62%——没过我们 80% 的中止线，任务正常完结。但 62% 这个数字本身很难看，我们把它记成"衍生债"，下一个迭代专门加了一个按国号前缀区分同名人物的 guard。这是个尴尬但诚实的数据点：**没触发的 Stop Rule 依然在产生信息**，它告诉你离悬崖还有多远。

**跨领域提示**：reject 率高意味着你的自动化层和领域知识还没磨合好。正确反应是停下来补 guard，而不是用更多人工 review 硬扛——后者不可持续，且会让专家疲劳到漏判。

## 类 5：跨迭代一致性类

```markdown
- 本轮成本 vs 最近一轮同类任务偏离 > 2x → Stop
  - 触发时：判断是数据复杂度变了，还是 pipeline 退化了

- 同类任务的 reject 率与历史基线偏离 > N% → Stop
  - 触发时：可能是 prompt 漂移 / 领域知识演化，需要显式决策

- 写入数与 dry-run 预测数不一致 → Stop
```

单轮指标正常不代表系统健康——温水煮青蛙式的漂移只能靠跨轮对比发现。

**实证**：开头那个 175 vs 179 的案例就属于这类，是我们触发次数最多的一条。那次的裁决是"接受 + 补透明度文档"：幂等去重是设计意图，但 dry-run 报告没把它讲清楚，于是给报告加了幂等拦截章节，并在复盘里记了一笔方法学缺陷。触发 Stop Rule 不丢人，**触发后留下完整决策记录，下次同样的偏差就有先例可查**。

**跨领域提示**：这类规则的前提是你留存了历史轮次的指标。从第一轮就开始记录成本、产出量、reject 率，第三轮起这类规则才有基线可用。

## 触发之后：标准处置协议

规则触发后 agent 该干什么，也要预先写死，否则"停下来"会退化成"慌乱地乱动"：

1. **立即停**——不要"再跑一个阶段看看"，也不要急着回滚（保留现场便于诊断）
2. **报告**——规则编号 + 触发数值 + 当前阶段状态 + 已完成/未完成的子任务
3. **等裁决**——人类从四个选项里选：A 接受并记债继续 / B 修因后重试 / C 回滚 / D 中止任务重立简报
4. **记录**——裁决写入独立文档，复盘时引用，便于跨迭代检索同类触发

## 4 个反模式

**反模式 1：模糊阈值。** ❌"成本不要太高就停" → ✅"成本 > $1.80 即停"。模糊规则等于把裁量权下放给了 agent，而 agent 的裁量永远倾向于继续跑。

**反模式 2：执行中"补"规则。** ❌ 任务跑到一半发现"应该有这条规则"，临时加上 → ✅ 只能写进下一轮的任务简报。中途加的规则没有经过冷静期，往往是对当前焦虑的过拟合；更糟的是它破坏了"规则先于执行"的契约——这个契约正是 agent 服从规则的基础。

**反模式 3："软"规则。** ❌"如果成本过高，请考虑停止" → ✅"成本 > $X，必须立即停止"。给 LLM 的指令里，"考虑""酌情""尽量"这类词等于没写。措辞必须是无条件命令式。

**反模式 4：违反但不报告。** ❌"$1.95 离 $1.80 也不算远，就继续了" → ✅ 触发必须上报、必须等裁决。这是四个反模式里最危险的一个：LLM 给"继续执行"编造合理化理由的能力是一流的，今天豁免 8%，下次就敢豁免 30%。豁免权必须收归人类。

## 给你的最小落地清单

不需要先搭一套治理框架，今天就能在自己的 agent 工作流里加上：

1. **给最贵的一步设一个具体数字的成本上限**，写进 system prompt 或任务简报：超过即停并上报，无例外。
2. **找出一个"绝不能为假"的断言**（哪怕只有一条），做成自动检查，agent 每次改完数据必须跑，非零即停。
3. **所有中止条件在任务开始前写下来**，执行中只许触发、不许新增或放宽。
4. **规定触发后的报告格式**：规则编号 + 触发数值 + 当前状态。明确告诉 agent：它没有豁免权。
5. **每次复盘记两个清单**——"触发了哪些"和"差点触发哪些"，用真实数据校准下一轮阈值。一个任务 5-8 条规则比较合适：少于 5 条覆盖不全，多于 10 条没人记得住。

Agent 时代的工程能力，一半在于让它跑起来，另一半在于敢放手让它跑——而敢放手的前提，是你提前想清楚了它必须停下来的全部条件。
