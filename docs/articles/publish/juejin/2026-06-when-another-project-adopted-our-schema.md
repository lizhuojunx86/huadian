# 另一个项目"抄"了我们的框架之后：4 条骨架活了下来，60% 的设计被扔掉

## 一、我们在做什么

过去一年多，我在做一个叫"华典智谱"（huadian）的开源项目：用一组分工明确的 AI agent（架构师、管线工程师、古籍专家、QA……各有职责边界和交接格式），严谨地、可审计地从古籍文本中构建结构化知识库。参考实现选的是《史记》——人物、事件、地点、世系，每一条抽取结论都能回溯到原文的具体段落。

做着做着，项目定位发生了一次转向：我们意识到真正有价值的不是"又一个史记数据库"，而是沉淀下来的那套**工程方法**——schema 怎么设计、溯源怎么连、消歧怎么做、人审怎么门控。于是项目转型为"知识库构建的工程框架"，史记降级为验证框架的案例。

但"框架"这个词有个尴尬之处：在被第二个项目用上之前，它只是你的一厢情愿。

## 二、然后，一个独立项目"抄"了它

2026 年 6 月，我们在做下游引用评估时，把一个叫 storyextractor 的项目拉出来冷读。这是一个目标完全不同的工具：从古籍里抽取"故事"和"事件"，做文言-白话对照语料，奔着产品去，不关心什么工程框架。

它的源码注释里写着这样几句（原文照录）：

- `model.py` 第 1 行：「mirrors huadian's ingest contract, adapted & lightweight」
- `db.py` 第 1 行：「Schema mirrors huadian's design (books / raw_texts / stories / story_segments), trimmed to a single-file local DB」
- `extract/entities.py`：「借 huadian identity_resolver 思路, 轻量化」

它的架构文档则明确声明：借用 huadian 的数据模型与管线设计，但**不并入** huadian、不受其框架约束——因为两个项目目标取向不同。

值得注意的是它的"借"法：**零代码 import，零数据复用**。它没有引 huadian 的任何包，没有用 huadian 的史记数据（自建了 916 段文白对照语料），数据库从 Postgres 换成了单文件 SQLite，第三方依赖砍到零。它借走的只有一样东西：**设计**。

### 先说丑话：这不是第三方采纳

诚实优先：**storyextractor 和 huadian 是同一个作者**。这不是"陌生人在 GitHub 上发现了你的框架"，而是同一个人在做第二个项目时，自发地复用了第一个项目的设计——并且自发地扔掉了大部分。

所以这份证据的成色是"准外部"：比框架作者在自己仓库里 dogfood 强（毕竟是独立项目、独立目标、独立技术选型，且明确拒绝并入），但弱于真正的第三方采纳。另外，两个项目都是古籍领域——**同域移植只能证明核心稳，不能证明跨域可移植**。这两条限定，本文后面还会回来。

但正因为是同一个作者、且第二个项目完全不在乎"维护框架的面子"，这次复用反而像一次诚实的实验：没有任何社交压力要求 storyextractor 保留 huadian 的设计。它保留的，是真觉得有用的；它扔掉的，是真觉得没用的。

## 三、活下来的 4 条骨架

逐字段对比两个项目的 schema，有四条设计契约在 storyextractor 里完整地"自发长出来"了——换了数据库（Postgres→SQLite）、换了抽取物（人物→故事/事件）、换了实现语言风格，这四条依然在。这是文章的干货部分，逐条讲。

### 契约 1：Source → Unit → Span 三层语料结构

源文本不是一团 blob。它应该被建模为三层：**源容器**（一本书）→ **章节单元**（一篇）→ **可寻址跨度**（一段）。第三层的稳定复合键 `(书, 章, 段号)` 就是整个系统的溯源锚点。

两边的字段几乎一一对应：huadian 是 `books(slug, title, author, dynasty, genre, license)` + `raw_texts(book_id, chapter, paragraph_no)`；storyextractor 是 `books`（同名同构）+ `raw_texts(book_id, chapter_seq, paragraph_no, original, vernacular)`。`db.py` 那句 "Schema mirrors huadian's design" 说的就是这一层。

这条看起来平平无奇，但它是后面三条的地基：没有段级稳定键，"溯源"就只能溯到"出自某本书"，等于没有。

### 契约 2：span 级溯源——指到跨度，不是指到文档

每条抽取结论必须能回溯到原文的**某个具体跨度**，而非泛泛的"出自某篇"。

huadian 的做法是字符偏移级：`source_evidences(raw_text_id, position_start, position_end, quoted_text)`，精确到第几个字。storyextractor 觉得太重，降成段范围级：`event_sources(chapter_seq, para_start, para_end, excerpt)`。

粒度变粗了，但结构同构——**粒度是可调的插件点，"抽取结论必须挂一条回原文的 span 指针"才是契约本身**。一个降级了精度却保留了指针的移植者，恰好帮你划清了核心与配置的边界。

### 契约 3：抽取层永带回指

实体、事件、故事这些抽取产物，**永远不脱离"它来自哪段"**：所有抽取表强制外键回到语料层。huadian 里 `persons`/`events` 经 `source_evidences` 锚定；storyextractor 里 `stories ← story_segments → raw_texts`，连地点表都带 `raw_text_id`。

这条契约的潜台词是一种数据完整性观：**抽取物可以错、可以删了重抽，但它与原文的链接不可断**。LLM 抽取是会出错的，错了要能定位、能重跑、能审计——前提是链接还在。链接一断，你的知识库就从"可审计的结构化语料"退化成"一堆来历不明的 JSON"。

### 契约 4：多源聚合，但逐源保留

这是四条里最有抽象价值的一条。纪传体史书有个天然特点：同一事件分散在多篇里——本纪记纲要，列传记细节，世家记旁证。朴素的做法是把多源信息 flatten 成一段融合文本，但这样**逐源的可审计性就丢了**：你再也说不清"这句话是哪篇说的"。

两个项目不约而同地选择了同一个结构：聚合实体 + 逐源记录。huadian 是 `events` + 多条 `event_accounts`（每条带自己的溯源，外加冲突表）；storyextractor 是 `events` + `event_sources`，每条来源带 span 指针和角色标签（主叙/详述/简述/评论/旁证），融合产物 `canonical_summary` 由 LLM 生成且要点标注来源。

**融合 ≠ 抹掉来源**。这一条在两个独立实现里都被有意地建模出来，是"这个模式真实存在、且领域无关"的最强证据。

## 四、被扔掉的 60%

现在说另一半故事——移植者扔掉了什么。粗略估算，storyextractor 砍掉了 huadian schema 的 60~70%：

逐条对照（huadian 的设计 → storyextractor 的处理）：

- **存储栈**：Postgres + pgvector + ORM 全套 → 9 张 SQLite 表，零第三方依赖
- **身份消歧**：六规则完整链 → "轻量化"为廉价确定性守卫 + LLM judge 两道门
- **溯源粒度**：字符偏移级 → 段范围级（粗粒度够用）
- **多语言文本**：简/繁/英 JSONB 结构 → 单字段白话译文
- **地理与纪年**：GIS 坐标 + 历史政区层级 + 纪年换算 → 现代地名 + 关键词，注释明示"永不生成沿革文本"
- **可信度建模**：可信度/出处双维度多级分级 → 砍成一个字段
- **事件冲突**：显式建模 → 省略

第一反应可能是心疼。但冷静下来看，这张"砍单"传递的信息比保留清单更刺耳也更有用：**要把这个框架搬到第二份语料上，第一步是扔掉它的大部分。**

被砍掉的东西有一个共同点：它们都是《史记》这个案例的**领域丰富度**——多语言是因为面向国际读者，GIS 是因为要画历史地图，纪年换算是因为古籍纪年混乱。它们没有一样是"语料抽取知识库"这个问题本身所要求的。

事后我们对自己的 schema 做了量化：**领域无关的核心只占约 1/3，其余 2/3 是史记案例的血肉**。而我们此前一直把这一整坨当作"框架"在打包。移植者用脚投票，把这个自欺戳穿了。

## 五、这对框架设计者意味着什么

### 教训一：你以为的框架，多半是你第一个案例的尸骸

框架从案例中长出来，这没问题——问题是你很难分清哪些是骨、哪些是这具案例的肉。我们的自测早就显示领域无关代码占比不达标，但直到一个移植者把 2/3 扔在地上，这个数字才变得无法回避。

对应的修正也很具体：框架的下一个版本只 ship 四条骨架构成的瘦核心；多语言、GIS、分级体系全部下放到 examples 目录作为案例层；消歧器提供"轻量档"（两道门）而非只有完整链——因为移植者已经实证了完整链对多数场景过重。storyextractor 那 9 张 SQLite 表，反而成了"核心应该多瘦"的天然参照实现。

### 教训二：模式传导 > 代码 import

这次复用全程零 import、零依赖、零数据搬运，传过去的只有设计契约。这逼我们重新理解"框架被采纳"是什么意思：**对一个工程方法类框架来说，最有生命力的传播单元不是包，是模式**。能被人看懂、记住、在自己的技术栈里用 9 张表重写出来的设计，比一个必须 `pip install` 的依赖更接近"可移植"的本义。

反过来这也是检验标准：如果你的框架只能靠 import 传播，砍掉包之后什么都剩不下，那它可能根本没有可言说的设计核心。

### 教训三：诚实标注证据的成色

最后再把丑话重复一遍，因为它决定了这篇文章的结论能走多远：

- **已证明**：四条骨架在同域（古籍叙事）、双实现（Postgres/SQLite）、双抽取目标（人物/故事）下字段级稳健。换执行者、换存储引擎，它们会自发长出来。
- **未证明**：跨域可移植性。两个实现都是古籍叙事文本。这四条契约在法律判例、财报、病历这类非叙事语料上是否成立，目前**没有任何证据**——这是下一步要做的实验，不是可以顺嘴宣称的结论。
- **成色**：同一作者的"准外部"验证，强于 dogfood，弱于第三方。

一个尴尬的发现（核心只占 1/3）加一条降级的证据（同域、准外部），听起来不如"我们的框架被成功采纳"响亮。但前者是可以据此做决策的工程事实，后者只是营销。做框架的人，最好早一点习惯前者的口感。
