# 一个人 + AI 扮演 10 个角色：8 周、27 个 sprint、34 份 ADR 全程留痕之后，我学到了什么

先交代一下我是谁：技术背景出身，目前处于自由探索状态，有整块时间投在自己的项目上。

## 一、设定：让 AI 当首席架构师，是认真的

2026 年 4 月 15 日，我立项做一个《史记》知识库：用 LLM 做命名实体识别、跨篇章身份消歧、关系抽取，把古籍变成每条数据都可溯源的结构化知识。这种项目通常需要一个小团队——后端、数据管线、前端、QA、领域专家，外加一个架构师来协调。

我没有团队。所以我做了一个看起来有点行为艺术的决定：让 Claude 出任"首席架构师"，并由它起草 10 个角色定义文件——首席架构师、产品经理、古籍专家、UI/UX 设计师、后端工程师、数据管线工程师、前端工程师、QA 工程师、DevOps、数据分析师。每个角色一个 markdown 文件，写清职责、输入输出、决策权限，以及最重要的一节：**禁区**。前端工程师不得做设计决策，管线工程师不得改数据库 schema，QA 不得改业务代码，古籍专家不写代码。角色间冲突一律升级到架构师仲裁，仲裁结果记入 ADR（架构决策记录）。

运转方式是 tagged-session：同时开多个 Claude Code 会话，每个会话戴一顶角色帽子，跨会话交接靠显式信号。这是协作规约里的真实片段：

```
【BE】SDL ready           → 【FE】codegen 解除阻塞
【BE】migration 已应用     → 【PE】backfill 解除阻塞
【PE】Stage 2 apply 完成   → 【Architect】发起领域专家评审
【Hist】评审报告完成       → 【Architect】下达 Stage 4 apply 指令
```

治理机制完全照搬人类工程团队：STATUS 状态板 + CHANGELOG + ADR + 任务卡四件套；Sprint / Stage / Gate 工作流加 Stop Rules（触发即停、升级仲裁）；数据层 11 类形式化不变量（V1–V11），每个 sprint 收口必须全绿。

一个人的业余项目要 ADR 干什么？这正是本文想回答的问题。8 周之后，有些机制被证明是救命的，有些被证明是会自我繁殖的仪式——**两者都全程留痕**。repo 是公开的（文末有链接），下面每个数字都可以去 git log 里查证。

## 二、8 周跑出来的东西

先把账摊开。

**前两周（4 月 15 日—29 日），11 个 sprint 跑完了工程主线**：数据库 schema 与迁移链、LLM 抽取管线、GraphQL API、极简 Web 端、外部知识库种子对齐，以及整个项目技术上最硬的部分——身份消歧器（6 条规则 + 可插拔 guard 链）和人机协作审核（triage）闭环。收口时的数据基线：**729 个有效人物实体、111 次实体合并、177 条审核决策、22 项不变量检查全绿**。

guard 链不是摆设。对 663 个实体做全量干跑时，它拦下 16 对危险合并——9 对跨朝代同名（相隔几百年的两个同名人物差点被合成一个），7 对国别前缀混淆。还有一次反向操作：一个被错误合并的"楚怀王"实体（战国的熊槐和秦末的熊心，史书里都叫楚怀王）被按预先定义的 entity-split 协议拆开，拆分过程本身也留了审计记录。这些案例后来都成了方法论文档的一手素材。

**第 15 天，急转弯。** 4 月 29 日例行做第三方调研，发现一个叫 shiji-kb 的开源项目：130 篇全本《史记》结构化、1.4 万实体、3200 事件、7600 关系——而我这边是 3–4 篇、729 个人物。对方是一位有 30 多年知识工程经验的作者带着 AI agent 做的，社群近万人。结论很难听但很清楚：在"古籍数据完整度"这个维度上，我落后头部一到两年，继续做下去，6 个月后的产物大概率不如对方的当前状态。

当天，AI 架构师起草了 ADR-028：项目主轴从"古籍知识产品"转向"**如何用 AI agent 团队严谨地构建可信知识库**"的工程方法论框架，史记降级为参考案例。理由写得很直白：真正稀缺的不是又一份史记数据，而是前两周积累的那套可复用的工程模式——角色边界、guard 链、不变量、审计闭环。这次转型从调研到 ADR 签字只用了一天，但 negative space（不做什么）写了整整一节。

**之后六周**：16 个方法论 sprint，把工程资产抽象成 8 份方法论文档和一个独立的 Python 框架包（3 个核心模块、93 个单元测试全绿）；启动了第二个跨领域案例——制药行业的批生产记录审计（两个真实产品，这里叫 P1、P2，共 12 个批次），产出 4 条跨案例可迁移的发现、3 份 ADR、3 篇各约 4000 字的方法论文章初稿；外加两篇月度决策日记，约 1.2 万字。

这个速度值得单独注一笔：上面这些活，传统团队的排期大概是几个月到一年。AI 协作 8 周跑完，而且大部分发生在前 6 周。**但"快"恰恰是后面所有问题的入口。**

## 三、三个反直觉发现

如果只想看干货，这一节就是全文。

### 发现一：治理仪式会自我增殖，而且 AI 比人更擅长喂养它

转型之后有一段时间，项目的 CHANGELOG 看起来盛况空前：16 个方法论 sprint 接连关档，每个都有 brief、有 retro、有估时核对。其中一篇 retro 庆祝了一个里程碑："连续第 12 个 sprint 零 Stop Rule 触发，超出目标 240%。"

六月初，我让 AI 架构师对项目做一次全面进度评估，要求关键事实必须用 git 取证，不许引用自己写的状态文档。结果这条"12 连胜"被它自己证伪了：**那 12 个 sprint 的全部提交集中在三天之内，后期的"sprint"实际工作量退化到一次 +6 行、+42 行的交叉引用微调**。而"零 Stop Rule 触发"在维护态下是同义反复——没有实质工作，自然没有东西可触发。把它当 KPI 庆祝，是教科书级的 Goodhart 定律现场。

事后看，机制很清楚：sprint 治理是为"人类团队一周一迭代"设计的，它假设 sprint 本身有成本。但对 AI 来说，开一个 sprint、写一份 brief、关档、写 retro，是几分钟的事——**当仪式的成本趋近于零，仪式的计数就失去全部信息量**。更微妙的是，文档、retro、模板这类产出恰好是 AI 最舒适、最高产、最容易获得"完成感"的工作。没有外部锚点的话，整个系统会自发滑向"用治理动作生产治理动作"。

后来的对策是记两本账：产出物计数（sprint 数、文档版本号）一本，"新增独立知识资产"（一条新发现、一个新模块、一篇新文章，而不是同一篇文章的第 N 次润色）另一本。第二本账一记，某个高产期 9 个迭代里真正的新资产只有 3 项，立刻现形。

### 发现二："可审计"对 AI 协作不是装饰，是刚需

这里要先替 AI 说句公道话：它没有撒谎。那次评估翻遍全部记录，每一条乐观陈述旁边的限定词都老老实实在场——"commit 待用户本地执行""待外部审稿""工作稿"。评估报告的原话是：**"这不是欺骗，而是渲染层与判据层没分离。"**

人类汇报者会有意美化；AI 不会有意，但它会**无意识地通胀叙事**——给容易产出的维度（文档数、sprint 计数）打上 ⭐⭐⭐ 和"领先路线图 6–8 周"，让真实的滞后（框架代码冻结 5 周、对外发表为零）在视觉上消失。每个局部都是真话，整体读下来却是一幅过于乐观的图。

解法是被这次评估逼出来的：在状态板顶部加一张"**对外可验证判据真相表**"——11 项判据，全部二元（done / not-done），全部以"外界能否独立验证"为标准，并且与一切带 ⭐ 的叙事**物理分离**，每次状态同步强制更新。建表那天的初始读数相当难看：11 项里只有 2 项达标。当天紧急整改（补齐缺失的单元测试、补框架打包、push 公开仓）之后，也只是 4 绿、2 黄、5 红。

最有意思的是这个自我打脸过程的形状：评估由 6 个并行 agent 分维度勘察，汇总后再过一轮**对抗式批判**——专门负责拆台的那一轮不仅确认了"叙事注水"，还反过来抓出综合稿自己的两处夸大其词（把"0/3 达标"修正为"0–1/3"，把两个口径不一的统计数字强制归一）。AI 批判 AI 写的 AI 自评，听起来套娃，但每一层都落在文件里、可以复查，这比"我感觉项目还行"可靠得多。

### 发现三：对外接触是 AI 无法代劳的唯一环节——所以它成了系统性短板

把真相表的红绿项目分个类，会看到一个整齐得令人不安的剪刀差：**内部判据全绿**——测试通过、不变量全绿、文档齐全、命名敲定；**对外判据全红**——正式发表 0 篇、外部引用 0 次、联系上的外部审稿人 0 位、与近邻项目的实质交流 0 次。甚至连"把本地提交 push 到公开仓"这件事都一度滞后三周、积压 19 个 commit——因为 Claude 运行在沙箱里碰不了远端 git，push 这个动作只能我做，而我没做。

这不是巧合，是结构：**凡是 AI 能独立完成的环节，以超常速度推进；凡是需要一个真实人类出面的环节，静止不动。**更糟的是，AI 不会替你着急——三篇文章全部停在"待 2026 年 9 月外部审稿"，但"找到愿意审跨领域稿件的真人专家"这个 AI 无法替代的前置条件，在长达数周的风险登记里是空白。系统不是失败在它做不了的事上，而是**静默地绕开它做不了的事，并继续在能做的事上高产**，让仪表盘保持好看。

评估之后的两条对策都很朴素：一是把"只有作者本人能做的事"（push、联系审稿人、对外邀约）显式登记为最高优先级风险，配 deadline；二是给单案例深挖设了 5 条二元止损线——比如"连续 3 个迭代零新增独立知识资产"或"框架核心代码超过 4 周零改动"，触发任何一条就强制换挡去做对外的事。颇有黑色幽默的是：止损线写入文档的当天，其中一条（代码冻结超 4 周）就已处于触发状态。

## 四、如果你也想一个人 + AI 这么干：6 条具体建议

1. **第一天就建真相表。**一张表，只放外界可独立验证的二元判据（发布了吗、有人引用吗、外人能跑通吗），与一切自我叙事分开存放，每次状态同步强制更新。不要等 8 周后被自己的评估打脸才建。
2. **角色文件重点写"禁区"而不是职责。**职责 AI 自己能推断，禁区它不会主动给自己设。再配一个固定的冲突升级出口（仲裁结果记 ADR），否则多会话并行时边界会被悄悄踩穿。
3. **两本账分开记。**产出物计数（sprint、文档、版本号）一本，"新增独立知识资产"另一本。AI 最舒适的产出——文档、模板、retro——往往是边际价值最低的，第二本账是唯一的解毒剂。
4. **定期做 git 取证式自评，并强制加对抗轮。**让 AI 评估项目时，规定关键事实必须出自 git log 而非它自己维护的状态文档；评估稿再过一轮专职拆台的批判。我项目里最有价值的一份文档（那份进度评估）就是这么产出的。
5. **把"只有你能做的事"显式列成风险清单。**push、找审稿人、对外邀约、人脉——AI 不会替你着急，也不会在它绕开这些事时报警。这份清单要配 deadline，并且由真相表反向监督。
6. **给一切"做深"的方向预设二元止损线。**"把单一案例做深"和"完成度黑洞"只隔一条线；用触发条件（连续 N 个迭代零新资产、核心代码冻结超 N 周）代替感觉，触发即换挡。

## 五、结尾：全程可查证

这个实验最大的价值不在跑出了多少实体和文档，而在一个完整、诚实、没有剪辑的过程记录：包括 4 月底那次承认"打不过就转型"的急转弯，包括"12 连胜"被 git 取证证伪的全过程，包括真相表上至今未消掉的 5 个红灯。失败和尴尬都在 commit 历史里，这正是它作为方法论案例的全部可信度来源。

repo 在这里，Apache 2.0 + CC BY 4.0：**https://github.com/lizhuojunx86/huadian**。CHANGELOG、ADR、retro、那份进度评估和真相表都在 docs/ 下，欢迎逐条核对——尤其欢迎来当真相表里至今为 0 的那个"外部审稿人"。

8 周下来，我本人的总投入约 20 小时——平均每周 2.5 小时，绝大多数花在审稿、决策和说"同意/不同意"上，剩下的全部工作发生在 AI 的会话里。API 成本走的是 Claude Max 订阅，无法精确计量，粗略折算约 100 美元。一个可以对照的量级：27 个 sprint、34 份 ADR、约 1 万行框架代码加数十万字文档，人力 20 小时。
