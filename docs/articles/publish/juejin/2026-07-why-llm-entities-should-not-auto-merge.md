# LLM 抽出来的实体，为什么不能自动合并——一套"提议 + 人审"的消歧工程方案

用 LLM 跨篇章抽取实体时，有一个几乎一定会撞上的问题：同一个人，会被抽成好几个"人"。

我在做一个《史记》知识库时遇到的真实情况是——"项羽"和"项籍"是同一个人，分散在不同篇章里被抽成了两条记录，该合并；但"楚怀王"在文本里其实指向两个不同的人（战国的熊槐和秦末的熊心），被错抽成了一条，该拆分。**合并和拆分这两件事，方向相反，而且代价极不对称。**

这篇文章讲的就是怎么处理这件事。它来自一个古籍项目，但方法是领域无关的——你在做法律案例、医疗档案、专利、任何需要跨文档把"同一个实体"认出来的知识库，都会碰到一样的结构。

## 第一原则：默认不自动合并

大多数人的第一直觉是：抽完之后，按名字归一，名字一样的合并掉。这是灾难的开始。

原因是**合并是破坏性且近乎不可逆的操作**。两条记录一旦合并成一个实体，它们的证据、关系、属性就缠在一起了；事后发现合错了要拆开，难度远大于当初不合。而"同名异人"在真实语料里极其常见——史记里同一个谥号（比如"文公""武王"）在不同诸侯国反复出现，按名字合并会把晋文公和别国的"文公"合成一个人。

所以这套系统的默认动作不是"合并"，而是：**提议合并，然后把提议推给人审**。规则永远只产出 proposal，不直接改数据。这一条看起来保守，但它把一个不可逆的破坏性操作，变成了一个可回退的建议——这是整个方案的地基。

## 规则是一道梯子，而且是"长出来的"

判断两条记录是不是同一个人，用的是一组按置信度排序的规则。真实的梯子大致是这样（置信度从高到低）：

- **表层名完全匹配**（最高，~0.95）：两条记录的主名一字不差
- **同义/异体字归一后匹配**（~0.90）：通过一个领域词典把异体字、通假字归一后相等
- **别名词典双向匹配**（~0.90）：一方的主名出现在另一方的别名集合里
- **基于身份注记的交叉引用**（最低，~0.65）：只是一个"可能是"的假说，置信度低，单独不足以定案

求值是 first-match-wins：命中高置信度规则就先走它。另外还有一条独立的前置通道——如果两条记录都已经锚定到同一个外部唯一标识（我们用的是 Wikidata 的 QID），那是最硬的证据，直接走前置 seed-match，不进后面的梯子。

这里有个反直觉但重要的经验：**这道规则梯子不是一次设计出来的，是长出来的。** 我们最初只有"表层名匹配"，跑了几篇之后发现一堆同名异人被合并、一堆异体字写法的同一个人没合并，才一条条加规则、一条条调置信度。如果你想一开始就设计出完美的规则集，大概率会过度工程化；更可行的是从最严的一两条起步，让真实数据告诉你下一条该加什么。

[作者补充：可在这里加一句你自己对"规则长出来"这件事最深的一次体感——比如哪条规则是被哪个具体的错误逼出来的]

## 光有规则不够：guard 是规则之后的否决层

规则会犯一类系统性错误：表层名匹配会把"晋文公"和另一个国家的"某文公"匹配上——名字确实一样，但根本不是一个人。

解决办法不是把规则改得更复杂，而是在规则之后加一个**独立的否决层（guard chain）**。规则说"我觉得这两条是同一个人"，guard 链逐条检查"有没有硬性理由否决这个提议"：

- **跨朝代 guard**：两条记录的活动年代差超过阈值（比如 200 年），直接否决——再像也不可能是同一个人
- **国别前缀 guard**：两个谥号一样但分属不同诸侯国，否决

关键设计在于：**guard 不是规则的一部分，是规则之后的一道独立关卡**。规则负责"找出疑似相同"，guard 负责"否决明显不可能"，两者职责分离。一个被 guard 拦下的提议，不会静默消失，而是变成一个标记状态（我们叫它 BlockedMerge），留痕、可复查。

## 每个提议都必须带证据

被推给人审的，不是一个光秃秃的"建议合并 A 和 B"，而是**附带证据的提议**：是哪条规则命中的、命中的依据是什么、有没有被哪个 guard 拦过。

这一条的价值在人审环节才显现。审的人不需要回到原始语料里重新调查，看一眼证据就能判断这个提议靠不靠谱。**没有证据的提议等于把调查成本又甩回给了人**，那样人审会迅速变成瓶颈。

## 人在环 + 不可变审计

最后一环：提议进入一个待审队列，人来做最终决策——批准合并、驳回、或者判定"其实该拆分"。而**决策一旦做出就不可变地写入审计表**，包括是谁、什么时候、基于什么证据做的判断。

为什么决策要不可变？因为知识库是会长期演化的资产，三个月后你可能想回查"当初为什么把这两条合了"。如果决策记录可以被随手覆盖，这个回查能力就没了。审计表不是合规摆设，它是让"几个月前的判断"在未来仍然可追溯的唯一办法。

## 给你的可迁移清单

如果你也在做跨文档的实体消歧，这套方案里真正可以拿走的是五条：

1. **默认不自动合并**——把不可逆的破坏性操作，降级成可回退的提议
2. **规则只产 proposal，不直接改数据**——规则是召回层，不是决策层
3. **guard 作为规则之后的独立否决层**——"找出疑似相同"和"否决明显不可能"职责分离
4. **每个提议强制带证据**——否则人审会变成瓶颈
5. **人的决策不可变留痕**——为知识库的长期可追溯性买单

这五条没有一条依赖古籍这个领域。它们解决的是同一个普适问题：**当机器的判断有可能错、而错误的代价不对称时，怎么把"机器的高效"和"人的可靠"接在一起，并且让整个过程事后还能查。**
