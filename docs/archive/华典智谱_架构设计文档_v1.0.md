## 一、架构总览

### 1.1 设计哲学

```
核心原则: "一次结构化，N次衍生"

原始古籍文本
    ↓ [AI结构化管线]
知识图谱 + 时空数据 + 事件库
    ↓ [统一API层]
    ├── 时空图谱引擎
    ├── 人物社交网络
    ├── 典故真相机
    ├── 决策案例库
    ├── 古人模拟器
    ├── 断案解谜游戏
    ├── 争议导航系统
    ├── 古地名翻译器
    ├── 情绪热力图
    ├── 供应链图谱
    ├── 写作灵感引擎
    └── ... 未来所有产品功能
```

所有产品功能共享同一套底层结构化数据。这是整个架构的灵魂——不是为每个功能单独建一套数据，而是建一个足够丰富的知识底座，让上层应用像查询视图一样自然生长。

### 1.2 四层架构

```
┌─────────────────────────────────────────────────────┐
│  L4: 应用层 (Applications)                           │
│  React/Next.js 前端 + 各产品模块                      │
├─────────────────────────────────────────────────────┤
│  L3: 服务层 (Services)                               │
│  GraphQL API Gateway + 领域微服务                     │
├─────────────────────────────────────────────────────┤
│  L2: 知识层 (Knowledge)                              │
│  知识图谱 + 向量库 + 时空索引 + 全文索引               │
├─────────────────────────────────────────────────────┤
│  L1: 数据层 (Data Pipeline)                          │
│  古籍摄入 → AI结构化 → 质量校验 → 入库                │
└─────────────────────────────────────────────────────┘
```

## 二、L1 数据层 — AI结构化管线

这是整个项目的地基。没有高质量的结构化数据，上层任何产品都是空中楼阁。

### 2.1 管线总流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  原始文本  │───→│  预处理   │───→│ AI抽取   │───→│ 质量校验  │───→│  入库     │
│  摄入     │    │  清洗     │    │ 结构化   │    │ 人工复核  │    │  索引     │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
```

### 2.2 原始文本摄入 (Ingestion)

**数据来源优先级**:

| 优先级 | 来源 | 格式 | 说明 |
|--------|------|------|------|
| P0 | 中华书局标点本电子版 | TXT/XML | 最权威，有标点分段 |
| P1 | 中国哲学书电子化计划 (ctext.org) | API/HTML | 开放获取，覆盖面广 |
| P1 | 维基文库 (zh.wikisource.org) | Wiki markup | 社区校对版 |
| P2 | 四库全书数据库 | 扫描PDF+OCR | 需OCR管线 |
| P3 | 地方志数据库 | 混合格式 | 结构差异大 |

**摄入数据模型**:

```typescript
interface RawText {
  id: string;                    // UUID
  source_id: string;             // 来源标识
  book_title: string;            // 书名 e.g. "史记"
  volume: string;                // 卷名 e.g. "项羽本纪"
  chapter: string;               // 章节
  paragraph_index: number;       // 段落序号
  raw_text: string;              // 原始文本
  text_version: string;          // 版本标识 e.g. "中华书局1959"
  dynasty: string;               // 成书朝代
  author: string;                // 作者
  metadata: {
    digitized_by: string;
    digitized_date: string;
    license: string;
    ocr_confidence?: number;
  };
}
```

### 2.3 AI结构化抽取 (Extraction)

这是核心管线。用LLM从原始文本中抽取8类结构化实体。

**抽取目标 — 8大实体类型**:

```typescript
// ① 人物实体
interface Person {
  id: string;
  names: {                       // 古人有多个称呼
    primary: string;             // 主要名称 e.g. "苏轼"
    courtesy: string;            // 字 e.g. "子瞻"
    art: string;                 // 号 e.g. "东坡居士"
    posthumous?: string;         // 谥号
    temple?: string;             // 庙号
    aliases: string[];           // 其他别称
  };
  dynasty: string;
  birth_year?: number;           // 公元纪年，BCE为负数
  death_year?: number;
  gender: string;
  roles: string[];               // e.g. ["诗人", "官员", "书法家"]
  faction?: string;              // 政治派系
  clan?: string;                 // 家族/郡望
}

// ② 地点实体
interface Place {
  id: string;
  ancient_names: {               // 古地名可能有多个
    name: string;
    dynasty: string;             // 该名称使用的朝代
    admin_level: string;         // 行政层级: 州/郡/县/关隘/山川
  }[];
  modern_name?: string;          // 今地名
  modern_province?: string;
  coordinates?: {                // WGS84坐标
    lat: number;
    lng: number;
  };
  geo_type: string;              // city/mountain/river/pass/battlefield
}

// ③ 事件实体
interface Event {
  id: string;
  title: string;                 // e.g. "鸿门宴"
  event_type: string;            // battle/political/cultural/economic/natural
  date: {
    start: HistoricalDate;
    end?: HistoricalDate;
    precision: "year" | "month" | "day" | "approximate";
  };
  location_ids: string[];
  participant_ids: string[];
  cause_event_ids: string[];     // 前因事件
  effect_event_ids: string[];    // 后果事件
  summary: string;               // AI生成的事件摘要
  significance: string;          // 历史意义
  tags: string[];                // e.g. ["军事", "权谋", "宴饮"]
}

// ④ 关系实体 (人物间)
interface Relationship {
  id: string;
  person_a_id: string;
  person_b_id: string;
  relation_type: string;         // 见下方关系类型表
  direction: "a_to_b" | "b_to_a" | "bidirectional";
  start_year?: number;
  end_year?: number;
  context: string;               // 关系背景说明
  evidence_ids: string[];        // 原文证据
}

// 关系类型枚举:
// political: 君臣/同僚/政敌/联盟/弹劾/举荐
// family: 父子/兄弟/夫妻/姻亲/族亲
// academic: 师生/同门/学术对手
// literary: 诗文酬唱/序跋推荐/文学批评
// military: 上下级/对手/同袍
// economic: 恩主/门客/商业伙伴

// ⑤ 制度实体
interface Institution {
  id: string;
  name: string;                  // e.g. "科举制"
  category: string;              // political/economic/military/legal/cultural
  dynasty_range: {
    start_dynasty: string;
    end_dynasty: string;
  };
  description: string;
  evolution: {                   // 制度演变链
    dynasty: string;
    change_description: string;
    year?: number;
  }[];
}

// ⑥ 物品/概念实体
interface Artifact {
  id: string;
  name: string;                  // e.g. "盐铁" "漕运" "丝绸"
  category: string;              // commodity/technology/concept/custom
  description: string;
  related_institution_ids: string[];
  trade_routes?: string[];       // 相关贸易路线
}

// ⑦ 典故实体
interface Allusion {
  id: string;
  name: string;                  // e.g. "卧薪尝胆"
  origin_event_id?: string;      // 关联的原始事件
  first_source_id: string;       // 最早出处
  original_meaning: string;
  modern_meaning: string;
  common_misunderstandings: string[];
  evolution: {                   // 含义演变
    era: string;
    meaning: string;
    source_id: string;
  }[];
}

// ⑧ 原文证据 (所有实体的根基)
interface SourceEvidence {
  id: string;
  raw_text_id: string;           // 关联到原始段落
  book_title: string;
  volume: string;
  quoted_text: string;           // 引用的原文片段
  entity_ids: string[];          // 该证据支撑的实体ID列表
  confidence: number;            // AI抽取置信度 0-1
  human_verified: boolean;       // 是否经人工校验
  version_note?: string;         // 版本差异说明
}
```

**HistoricalDate 特殊处理**:

```typescript
interface HistoricalDate {
  year: number;                  // 公元纪年，BCE用负数
  month?: number;
  day?: number;
  reign_era?: string;            // 年号 e.g. "贞观"
  reign_year?: number;           // 年号纪年 e.g. 3 (贞观三年)
  lunar_month?: number;          // 农历月
  lunar_day?: number;
  sexagenary_year?: string;      // 干支纪年
}
```

### 2.4 AI抽取管线详细设计

```
每个原始段落 (RawText)
    │
    ├─ Step 1: NER抽取
    │   Prompt: 从该段落中识别所有人物、地点、时间、官职、物品
    │   Model: Claude Sonnet (性价比最优)
    │   Output: 实体列表 + 在原文中的位置
    │
    ├─ Step 2: 关系抽取
    │   Prompt: 基于已识别实体，抽取实体间的关系
    │   Input: 原文 + Step1的实体列表
    │   Output: Relationship列表
    │
    ├─ Step 3: 事件抽取
    │   Prompt: 识别该段落描述的事件，提取因果链
    │   Output: Event实体 + cause/effect链接
    │
    ├─ Step 4: 实体消歧 & 合并
    │   问题: "高祖" 在不同史书中指不同人
    │   方案: 基于上下文(书名+卷名+朝代)进行消歧
    │   工具: 向量相似度 + 规则引擎混合
    │
    ├─ Step 5: 地理编码
    │   古地名 → 现代坐标
    │   数据源: CHGIS (中国历史地理信息系统)
    │   回退: LLM推理 + 人工标注
    │
    └─ Step 6: 质量校验
        ├─ 自动: 实体一致性检查、时间逻辑校验
        ├─ 采样: 随机抽取5%由人工复核
        └─ 众包: 开放社区校对接口 (Phase 3)
```

### 2.5 管线技术选型

```yaml
Pipeline Orchestration:
  engine: Prefect / Temporal     # 工作流编排
  queue: Redis Streams           # 任务队列
  why: 支持重试、断点续跑、进度追踪

LLM Calls:
  primary: Claude Sonnet         # 结构化抽取主力
  fallback: Claude Haiku         # 简单分类任务
  framework: LangChain / 自研    # Prompt管理 + 输出解析
  batch: Anthropic Batch API     # 批量处理降本50%

Storage:
  raw_texts: PostgreSQL          # 原始文本存储
  extracted: PostgreSQL + JSONB  # 结构化实体
  vectors: pgvector              # 向量检索(同一个PG实例)
  files: MinIO / S3              # 大文件(扫描件等)

Processing:
  language: Python 3.12+
  async: asyncio + httpx         # 并发LLM调用
  validation: Pydantic v2        # 数据模型校验
```

### 2.6 管线配置与成本估算

```yaml
# 以《史记》为例 (约52万字, 130篇, ~5200段落)

```

## 三、L2 知识层 — 存储与索引

### 3.1 核心数据库设计

采用 **PostgreSQL** 作为单一数据库引擎，利用其扩展能力统一处理关系数据、图查询、向量检索和全文搜索。

**为什么不用专门的图数据库(Neo4j)或向量数据库(Pinecone)?**
- 项目初期数据规模在百万级实体以内，PG完全能胜任
- 减少运维复杂度，一个数据库搞定四种查询模式
- 后期如需扩展，可以平滑迁移特定模块

```sql
-- ========================================
-- 核心Schema设计
-- ========================================

-- 启用扩展
CREATE EXTENSION IF NOT EXISTS pgvector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;      -- 模糊搜索
CREATE EXTENSION IF NOT EXISTS btree_gist;   -- 时间范围索引

-- ----------------------------------------
-- 原始文本表
-- ----------------------------------------
CREATE TABLE raw_texts (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_id       TEXT NOT NULL,
    book_title      TEXT NOT NULL,
    volume          TEXT,
    chapter         TEXT,
    paragraph_index INTEGER,
    raw_text        TEXT NOT NULL,
    text_version    TEXT,
    dynasty         TEXT,
    author          TEXT,
    metadata        JSONB DEFAULT '{}',
    embedding       vector(1536),            -- 语义向量
    created_at      TIMESTAMPTZ DEFAULT now(),

    -- 全文搜索
    tsv             tsvector GENERATED ALWAYS AS (
        to_tsvector('simple', raw_text)
    ) STORED
);

CREATE INDEX idx_raw_texts_tsv ON raw_texts USING GIN(tsv);
CREATE INDEX idx_raw_texts_embedding ON raw_texts USING ivfflat(embedding vector_cosine_ops);
CREATE INDEX idx_raw_texts_book ON raw_texts(book_title, volume, paragraph_index);

-- ----------------------------------------
-- 人物表
-- ----------------------------------------
CREATE TABLE persons (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name_primary    TEXT NOT NULL,
    name_courtesy   TEXT,                    -- 字
    name_art        TEXT,                    -- 号
    name_posthumous TEXT,                    -- 谥号
    name_temple     TEXT,                    -- 庙号
    aliases         TEXT[] DEFAULT '{}',
    dynasty         TEXT,
    birth_year      INTEGER,                 -- BCE为负数
    death_year      INTEGER,
    gender          TEXT DEFAULT 'male',
    roles           TEXT[] DEFAULT '{}',
    faction         TEXT,
    clan            TEXT,
    biography       TEXT,                    -- AI生成的人物小传
    embedding       vector(1536),
    metadata        JSONB DEFAULT '{}',

    -- 全名搜索索引
    all_names       TEXT GENERATED ALWAYS AS (
        COALESCE(name_primary, '') || ' ' ||
        COALESCE(name_courtesy, '') || ' ' ||
        COALESCE(name_art, '') || ' ' ||
        COALESCE(name_posthumous, '') || ' ' ||
        COALESCE(name_temple, '') || ' ' ||
        array_to_string(aliases, ' ')
    ) STORED
);

CREATE INDEX idx_persons_names ON persons USING GIN(all_names gin_trgm_ops);
CREATE INDEX idx_persons_dynasty ON persons(dynasty);
CREATE INDEX idx_persons_years ON persons(birth_year, death_year);

-- ----------------------------------------
-- 地点表
-- ----------------------------------------
CREATE TABLE places (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    modern_name     TEXT,
    modern_province TEXT,
    geo_type        TEXT,                    -- city/mountain/river/pass/battlefield
    coordinates     GEOMETRY(Point, 4326),   -- PostGIS
    metadata        JSONB DEFAULT '{}'
);

CREATE INDEX idx_places_geo ON places USING GIST(coordinates);

-- 古地名映射表 (一个地点在不同朝代有不同名称)
CREATE TABLE place_names (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    place_id        UUID REFERENCES places(id),
    ancient_name    TEXT NOT NULL,
    dynasty         TEXT NOT NULL,
    admin_level     TEXT,                    -- 州/郡/县/关隘
    year_start      INTEGER,
    year_end        INTEGER,

    UNIQUE(place_id, ancient_name, dynasty)
);

CREATE INDEX idx_place_names_search ON place_names USING GIN(ancient_name gin_trgm_ops);

-- ----------------------------------------
-- 事件表
-- ----------------------------------------
CREATE TABLE events (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title           TEXT NOT NULL,
    event_type      TEXT,                    -- battle/political/cultural/economic/natural
    year_start      INTEGER,
    year_end        INTEGER,
    month_start     INTEGER,
    day_start       INTEGER,
    date_precision  TEXT DEFAULT 'year',     -- year/month/day/approximate
    reign_era       TEXT,                    -- 年号
    reign_year      INTEGER,
    summary         TEXT,
    significance    TEXT,
    tags            TEXT[] DEFAULT '{}',
    embedding       vector(1536),
    metadata        JSONB DEFAULT '{}'
);

CREATE INDEX idx_events_year ON events(year_start, year_end);
CREATE INDEX idx_events_type ON events(event_type);
CREATE INDEX idx_events_tags ON events USING GIN(tags);

-- 事件-地点关联
CREATE TABLE event_places (
    event_id    UUID REFERENCES events(id),
    place_id    UUID REFERENCES places(id),
    role        TEXT,                        -- 发生地/出发地/目的地
    PRIMARY KEY (event_id, place_id)
);

-- 事件-人物关联
CREATE TABLE event_participants (
    event_id    UUID REFERENCES events(id),
    person_id   UUID REFERENCES persons(id),
    role        TEXT,                        -- 主导者/参与者/受害者/旁观者
    PRIMARY KEY (event_id, person_id)
);

-- 事件因果链
CREATE TABLE event_causality (
    cause_event_id  UUID REFERENCES events(id),
    effect_event_id UUID REFERENCES events(id),
    relation_type   TEXT DEFAULT 'causes',   -- causes/enables/prevents
    confidence      REAL DEFAULT 0.8,
    PRIMARY KEY (cause_event_id, effect_event_id)
);

-- ----------------------------------------
-- 人物关系表
-- ----------------------------------------
CREATE TABLE relationships (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    person_a_id     UUID REFERENCES persons(id),
    person_b_id     UUID REFERENCES persons(id),
    relation_type   TEXT NOT NULL,
    relation_subtype TEXT,
    direction       TEXT DEFAULT 'bidirectional',  -- a_to_b/b_to_a/bidirectional
    start_year      INTEGER,
    end_year        INTEGER,
    context         TEXT,
    metadata        JSONB DEFAULT '{}'
);

CREATE INDEX idx_relationships_persons ON relationships(person_a_id, person_b_id);
CREATE INDEX idx_relationships_type ON relationships(relation_type);

-- ----------------------------------------
-- 制度表
-- ----------------------------------------
CREATE TABLE institutions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            TEXT NOT NULL,
    category        TEXT,
    start_dynasty   TEXT,
    end_dynasty     TEXT,
    start_year      INTEGER,
    end_year        INTEGER,
    description     TEXT,
    embedding       vector(1536),
    metadata        JSONB DEFAULT '{}'
);

-- 制度演变记录
CREATE TABLE institution_changes (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    institution_id  UUID REFERENCES institutions(id),
    dynasty         TEXT,
    year            INTEGER,
    change_description TEXT,
    sort_order      INTEGER
);

-- ----------------------------------------
-- 典故表
-- ----------------------------------------
CREATE TABLE allusions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            TEXT NOT NULL,
    origin_event_id UUID REFERENCES events(id),
    original_meaning TEXT,
    modern_meaning  TEXT,
    common_misunderstandings TEXT[] DEFAULT '{}',
    embedding       vector(1536),
    metadata        JSONB DEFAULT '{}'
);

-- 典故含义演变
CREATE TABLE allusion_evolution (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    allusion_id     UUID REFERENCES allusions(id),
    era             TEXT,
    meaning         TEXT,
    sort_order      INTEGER
);

-- ----------------------------------------
-- 物品/概念表
-- ----------------------------------------
CREATE TABLE artifacts (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            TEXT NOT NULL,
    category        TEXT,
    description     TEXT,
    trade_routes    TEXT[] DEFAULT '{}',
    embedding       vector(1536),
    metadata        JSONB DEFAULT '{}'
);

-- ----------------------------------------
-- 原文证据表 (所有实体的溯源根基)
-- ----------------------------------------
CREATE TABLE source_evidences (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    raw_text_id     UUID REFERENCES raw_texts(id),
    quoted_text     TEXT NOT NULL,
    book_title      TEXT NOT NULL,
    volume          TEXT,
    confidence      REAL DEFAULT 0.8,
    human_verified  BOOLEAN DEFAULT FALSE,
    version_note    TEXT,
    created_at      TIMESTAMPTZ DEFAULT now()
);

-- 证据-实体多对多关联 (通用)
CREATE TABLE evidence_links (
    evidence_id     UUID REFERENCES source_evidences(id),
    entity_type     TEXT NOT NULL,           -- person/event/relationship/allusion/...
    entity_id       UUID NOT NULL,
    PRIMARY KEY (evidence_id, entity_type, entity_id)
);

CREATE INDEX idx_evidence_links_entity ON evidence_links(entity_type, entity_id);

-- ----------------------------------------
-- 史料版本差异表 (争议导航系统的基础)
-- ----------------------------------------
CREATE TABLE version_conflicts (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    entity_type     TEXT NOT NULL,
    entity_id       UUID NOT NULL,
    conflict_type   TEXT,                    -- factual/interpretive/chronological/omission
    description     TEXT,
    source_a_id     UUID REFERENCES source_evidences(id),
    source_b_id     UUID REFERENCES source_evidences(id),
    analysis        TEXT,                    -- AI分析
    metadata        JSONB DEFAULT '{}'
);
```

**图遍历示例 (人物关系N度搜索)**:

```sql
-- 查找苏轼的2度社交网络
WITH RECURSIVE social_graph AS (
    -- 起点
    SELECT person_b_id AS person_id, relation_type, 1 AS depth
    FROM relationships
    WHERE person_a_id = :sushi_id

    UNION ALL

    -- 递归展开
    SELECT r.person_b_id, r.relation_type, sg.depth + 1
    FROM relationships r
    JOIN social_graph sg ON r.person_a_id = sg.person_id
    WHERE sg.depth < 2
)
SELECT DISTINCT p.name_primary, sg.relation_type, sg.depth
FROM social_graph sg
JOIN persons p ON p.id = sg.person_id;
```

**为什么选GraphQL?**
- 前端需要高度灵活的数据组合 (人物+关系+事件+地点一次查回)
- 避免REST的N+1请求问题
- 各产品模块可以自由组合字段，不需要后端为每个视图写接口

```markdown
# CLAUDE.md — HuaDian (华典智谱) 项目指令

## 项目概述
华典智谱是一个中国古籍AI知识平台，将古籍文本转化为可探索、可交互的知识系统。

## 技术栈
- **前端**: Next.js 14+ (App Router), TypeScript, Tailwind CSS, shadcn/ui
- **后端API**: Node.js + TypeScript, GraphQL (Yoga/Apollo), Drizzle ORM
- **数据管线**: Python 3.12+, Prefect, Anthropic SDK
- **数据库**: PostgreSQL 16 (pgvector, PostGIS, pg_trgm)
- **缓存**: Redis 7
- **部署**: Docker Compose
- **包管理**: pnpm workspace + Turborepo

## 项目结构
- `apps/web/` — Next.js前端
- `services/api/` — GraphQL API服务
- `services/pipeline/` — Python数据管线
- `packages/shared-types/` — 共享TypeScript类型
- `packages/db-schema/` — 数据库schema
- `db/init/` — SQL初始化脚本

## 开发规范
- 所有代码使用TypeScript (前后端) / Python (管线)
- 前端组件使用函数式组件 + Hooks
- GraphQL schema-first开发: 先改schema，再写resolver
- 数据库变更必须通过migration
- 每个PR必须包含相关测试
- Commit message使用中文，格式: `[模块] 描述`

## 常用命令
```bash
pnpm install              # 安装依赖
pnpm dev                  # 启动开发服务器
docker compose up -d      # 启动数据库+Redis
pnpm db:migrate           # 运行数据库迁移
pnpm db:seed              # 导入种子数据

# 管线
cd services/pipeline
python -m src.cli ingest --source ctext --book 史记
python -m src.cli extract --book 史记 --steps ner,relations,events
python -m src.cli validate --book 史记
```

## 数据模型核心
8大实体: Person, Place, Event, Relationship, Institution, Artifact, Allusion, SourceEvidence
所有实体通过 SourceEvidence 溯源到原文。
详见 docs/data-model.md

## AI调用规范
- 结构化抽取用 Claude Sonnet (性价比)
- 实时对话/模拟用 Claude Sonnet
- 简单分类用 Claude Haiku
- 所有LLM调用通过 ai/gateway.ts 统一管理
- Prompt模板集中在 prompts/ 目录
- 必须有输出格式校验 (Pydantic / Zod)

## 十、关键技术决策记录 (ADR)

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| ADR-001 | 数据库 | PostgreSQL + 扩展 | 一个DB覆盖关系/图/向量/地理/全文，降低运维复杂度 |
| ADR-002 | API协议 | GraphQL | 前端需要灵活组合多实体数据，避免REST的N+1问题 |
| ADR-003 | 前端框架 | Next.js | SSR/SSG对SEO关键，古籍内容需要被搜索引擎索引 |
| ADR-004 | 管线语言 | Python | NLP/数据处理生态最成熟，Anthropic SDK原生支持 |
| ADR-005 | API语言 | TypeScript/Node | 与前端共享类型，GraphQL生态成熟 |
| ADR-006 | Monorepo | pnpm + Turborepo | 前后端类型共享，统一构建流程 |
| ADR-007 | LLM策略 | Sonnet为主+Haiku辅助 | 结构化抽取需要Sonnet级别能力，分类用Haiku降本 |
| ADR-008 | 地图引擎 | MapLibre GL | 开源免费，性能好，支持自定义瓦片 |
| ADR-009 | ORM | Drizzle | 类型安全，轻量，原生SQL能力强 |
| ADR-010 | 部署 | Docker Compose → K8s | MVP阶段简单部署，后期按需迁移 |
