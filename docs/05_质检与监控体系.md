# 05 质检与监控体系

> 本文件规定"**从底层数据到前端界面全链路嵌入式**"的质检体系与监控体系。
> 质检（QC）的目的：确保数据和处理结果**有效、稳定**。
> 监控（Observability + Analytics）的目的：为未来做成 APP 时理解用户行为、做 A/B 测试、反馈闭环。
>
> 两者共享基础设施（OpenTelemetry + 事件总线），但关注点不同：
> - 质检 = 系统对自己负责
> - 监控 = 产品对用户负责

---

## 一、设计原则

1. **嵌入式 > 事后**：质检必须是每层的"原生 hook"，不是定期跑的独立脚本
2. **阻断式 vs 观察式**：关键路径质检失败应**阻断下游**（如不合格数据不入库）；观察式监控只记录不阻断
3. **全链路 trace_id**：一次用户请求从前端到 LLM 抽取结果到 DB 的每一跳都带同一个 trace_id
4. **可反向追溯**：用户"存疑" → evidence_id → prompt_version → pipeline_run → llm_call → 原文段落，六步内追到根
5. **最小埋点原则**：埋点清单是有限集合，新增必须走 RFC

## 二、五层质检体系

```
┌──────────────────────────────────────────┐
│ L5 内容层：人工抽样 / 专家标注 / 社区反馈  │   ← 真人 + 专家
├──────────────────────────────────────────┤
│ L4 前端层：渲染错误 / 可访问性 / 可达性   │   ← Sentry + Lighthouse
├──────────────────────────────────────────┤
│ L3 API 层：schema / 响应 / 延迟 / N+1    │   ← GraphQL inspector + OTel
├──────────────────────────────────────────┤
│ L2 管线层：LLM 输出 / 抽取置信度 / 格式  │   ← TraceGuard
├──────────────────────────────────────────┤
│ L1 数据层：一致性 / 完整性 / 外键 / 去重 │   ← Postgres 约束 + 定时质检
└──────────────────────────────────────────┘
```

### L1 数据层质检

**嵌入机制**：
- Postgres 约束（NOT NULL / UNIQUE / CHECK / FK）是第一道
- `CHECK` 约束示例：
  ```sql
  CHECK (birth_year IS NULL OR death_year IS NULL OR birth_year <= death_year)
  CHECK (year_start IS NULL OR year_end IS NULL OR year_start <= year_end)
  CHECK (reality_status IN ('historical','legendary','mythical','fictional','composite','uncertain'))
  CHECK (provenance_tier IN ('primary_text','scholarly_consensus','ai_inference','community_contribution','expert_annotation'))
  ```
- 触发器：`entity_revisions` 自动记录修订

**定时质检规则**（`services/qc/rules/*.py`）：
| 规则 ID | 描述 | 阈值 |
|---------|------|------|
| QC-D01 | 孤儿外键检测 | 0 条 |
| QC-D02 | 实体无任何 evidence 比例 | < 5% |
| QC-D03 | 人物死亡早于出生 | 0 条 |
| QC-D04 | 事件时间区间非法（start > end） | 0 条 |
| QC-D05 | place.geometry 无效几何 | 0 条 |
| QC-D06 | 同 slug 重复 | 0 条 |
| QC-D07 | raw_text 未 resolve mentions 比例 | < 10% |
| QC-D08 | reign_era 跨 polity 日期冲突 | 0 条 |
| QC-D09 | 多语言字段缺失 zh-Hans | 0 条 |

**运行**：每日凌晨定时 + 每次管线完成后触发

## 四、监控与埋点体系

### 4.1 目标

- 了解核心功能使用率（哪些实体被访问、哪些模块热门）
- 评估内容质量（停留时长、证据展开率、反馈率）
- 支持 A/B 测试（比较不同视觉、不同推荐算法）
- 反馈闭环（用户质疑 → 追溯 → 修复）

### 4.2 事件字典（核心埋点）

```typescript
// packages/shared-types/src/analytics.ts

type AnalyticsEvent =
  // 浏览
  | { type: 'page_view'; path: string; params: object }
  | { type: 'entity_view'; entity_type: string; entity_id: string; slug: string }
  | { type: 'evidence_expand'; evidence_id: string; entity_id: string }
  | { type: 'evidence_collapse'; evidence_id: string }

  // 搜索
  | { type: 'search_submit'; query: string; filters: object }
  | { type: 'search_result_click'; query: string; result_id: string; rank: number }
  | { type: 'search_no_result'; query: string }

  // 地图
  | { type: 'map_pan'; bounds: GeoBounds }
  | { type: 'map_zoom'; zoom_level: number }
  | { type: 'map_layer_toggle'; layer: string; enabled: boolean }
  | { type: 'timeline_scrub'; year_start: number; year_end: number }

  // 交互
  | { type: 'graph_node_click'; node_id: string; graph_type: string }
  | { type: 'timeline_event_click'; event_id: string }
  | { type: 'allusion_truth_expand'; allusion_id: string }

  // 模拟器 / 游戏
  | { type: 'simulation_start'; scenario: string; persona_id: string }
  | { type: 'simulation_action'; session_id: string; action_type: string }
  | { type: 'simulation_end'; session_id: string; duration_ms: number; outcome: string }
  | { type: 'investigation_start'; case_id: string }
  | { type: 'investigation_hypothesis_submit'; case_id: string; hypothesis: string }

  // 反馈
  | { type: 'feedback_open'; entity_type: string; entity_id: string }
  | { type: 'feedback_submit'; feedback_id: string; feedback_type: string }

  | { type: 'experiment_exposure'; experiment_id: string; variant: string }
```

每个事件自动附带：`user_id / anon_session_id / trace_id / timestamp / client_info`

```yaml

  Logs: Loki
```

### 4.5 埋点接入规范

**规则**：
1. 所有新组件必须在设计评审时明确埋点需求
2. 埋点字典必须在 `packages/shared-types/src/analytics.ts` 中定义
3. 禁止在组件内拼接 event payload；统一通过 `useAnalytics()` hook
4. 后端所有 GraphQL resolver 自动接入 OTel trace
5. 敏感字段（用户输入、反馈文本）禁止直接入 analytics；只存哈希或枚举

**接入模板（前端）**：
```tsx
const { track } = useAnalytics();
<Button onClick={() => {
  track({ type: 'evidence_expand', evidence_id: ev.id, entity_id: entity.id });
  openEvidence(ev);
}}>查看原文</Button>
```

## 五、反馈闭环（最重要的闭合）

这是把"监控"与"质检"连起来的核心机制，也是产品长期质量的保证。

### 5.1 链路图

```
┌────────────┐       ┌────────────┐
│ 用户阅读某  │  看到 │ 存疑 / 反馈 │
│ 实体信息   │──────→│   按钮     │
└────────────┘       └─────┬──────┘
                           │
                 收集 entity_id + evidence_id
                 + prompt_version + model_version
                           │
                           ▼
                    ┌────────────┐
                    │ feedback   │
                    │  table     │
                    └─────┬──────┘
                          │
          ┌───────────────┴────────────────┐
          ▼                                ▼
  ┌──────────────┐              ┌──────────────────┐
  │ QA 看板      │              │ Analytics 分析   │
  │ 分类 / 优先级 │              │ 反馈率 / 热点    │
  └──────┬───────┘              └──────────────────┘
         │
    归因分析：是数据错？prompt 错？模型错？
         │
         ▼
  ┌──────────────────────────┐
  │ 选择修复路径：            │
  │ - 修 raw_text → 重跑      │
  │ - 改 prompt → 重跑        │
  │ - 换模型 → 重跑           │
  │ - 人工直接修 entity       │
  └─────┬────────────────────┘
        │
        ▼
  记 ADR + 更新 entity_revisions + 通知反馈用户
```

### 5.2 反馈字段（已在 docs/02 feedback 表中定义）

关键是 **一次反馈能回溯到**：
- `entity_id` + `entity_type`：哪个实体被质疑
- `evidence_id`：关联的证据
- `prompt_version`：抽取时用的 prompt 版本
- `model_version`：抽取时用的模型版本
- `trace_id`：那次请求的完整调用链

### 5.3 QA 看板

`/admin/qa/feedback` 页面（Phase 2）：
- 按 `resolved=false` 排序
- 聚合：每周反馈数、热点实体 Top 10、问题类型分布
- 分派：指派到 `historian / pipeline / backend`
- 处理动作：`dismiss / fix_and_rerun / flag_as_known_issue`

## 六、A/B 测试框架

### 6.1 最小可用设计

```sql
CREATE TABLE experiments (
  id              UUID PRIMARY KEY,
  key             TEXT UNIQUE NOT NULL,    -- 'search_ranking_v2'
  description     TEXT,
  variants        JSONB,                    -- [{'key':'control','weight':50}, {'key':'treatment','weight':50}]
  status          TEXT,                     -- draft / running / paused / ended
  start_at        TIMESTAMPTZ,
  end_at          TIMESTAMPTZ,
  target_metric   TEXT,                     -- 'evidence_expand_rate'
  created_by      TEXT
);

CREATE TABLE experiment_exposures (
  anon_id         TEXT NOT NULL,
  experiment_key  TEXT NOT NULL,
  variant         TEXT NOT NULL,
  exposed_at      TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (anon_id, experiment_key)
);
```

### 6.2 变体分配

```typescript
// 稳定 hash：同一 anon_id 始终落入同一变体
function variantOf(anonId: string, experiment: Experiment): string {
  const hash = murmur3(`${experiment.key}:${anonId}`);
  return weightedPick(experiment.variants, hash);
}
```

### 6.3 使用模式
- 用 feature flag 在后端/前端分支
- 曝光自动打点 `experiment_exposure`
- 转化事件匹配 `target_metric`
- 每晚跑显著性分析脚本

## 十二、D-route Invariant Pattern 抽象（2026-04-29 新增）

V1-V11 不只是史记数据的 invariants — 也是 **Agentic KE 框架** 的 Layer 1 核心抽象资产。任何 KE 项目可以沿用同一组 invariant 设计 pattern。

### 12.1 Invariant 五大类（领域无关）

| Pattern | 描述 | 史记案例 | 跨领域示例 |
|---------|------|---------|----------|
| Upper-bound | 某属性的上限约束 | V1: is_primary ≤ 1 per person | 法律：一份合同至多 1 个 primary 当事方 |
| Lower-bound | 某属性的下限约束 | V9: ≥ 1 primary per active person | 医疗：每个病例至少 1 个主诊断 |
| Containment | A 集合必须包含于 B 集合 | V8: 短名 prefix containment | 佛经：术语短形必须包含于全称集合 |
| Orphan | 无下游引用的孤立行检测 | V10 子规则: orphan target | 专利：无引用专利的孤立检测 |
| Cardinality | 关联表的行数边界 | V11: anti-ambiguity 卡方 | 任意：N:M 关联的边界检测 |

### 12.2 Invariant 设计契约（C-23 应用）

新增任何 invariant 必须满足：

1. **领域无关接口**：Invariant 函数签名不含 domain-specific 类型（用 generic `Entity`, `Relation` 等）
2. **domain-specific 参数显式化**：领域规则 = 配置文件 / yaml / json 注入，不硬编码
3. **失败可追溯**：违反时必须输出违反行的标识符（如 person.id），不能只说"有错"
4. **测试覆盖 ≥ 3 case**：positive / negative / boundary 各 1
5. **触发时机文档化**：每次 sprint 收口 / 每次相关代码合入 / 等

### 12.3 跨领域 Invariant 设计指南（草稿）

为新领域设计 V1-VN invariants 时：

1. 先列出该领域的"数据正确性铁律"（业务专家口述 → 工程师形式化）
2. 用五大类 pattern 套（5/5 都套不进的 invariant 多半是 QC 规则不是 invariant）
3. 写 invariant 函数 + ≥ 3 测试 case
4. 集成到 sprint gate（每次 sprint 收口必跑）
5. 违反时纳入 stop rule（不阻塞当前 sprint 完成；阻塞下游 sprint 启动）

详细 pattern catalog + cross-domain 设计指南见 [docs/methodology/04-invariant-pattern.md](methodology/04-invariant-pattern.md)（待 Stage C 起草）。
