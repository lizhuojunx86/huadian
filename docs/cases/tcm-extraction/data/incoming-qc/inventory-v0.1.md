## 3. 检验报告样本结构（基于甘草饮片 L1 PDF）

每份检验记录 PDF 通常 **9-56 页**，结构分四段：

### 3.1 主页 1-3（高密度数据）

- **第 1 页 — 检验报告**：报告编号 / 品名 / 批号 / 规格 / 供货单位 / 检验依据 / 来货数量 / 检验日期 / 报告日期 + 检验项目 × 标准 × 结果 × 结论 全表 + 总结论 + 签名（报告人/复核人/批准人）+ 质量保证部红章
- **第 2 页 — 物料放行单**：质量保证部审核员 + 部长 + 受权人三级签批，是 QA 的最终决策记录
- **第 3 页 — 请验单**：仓储部发起的检验请求，含厂家批号 ↔ 进厂批号 映射

这 3 页是机打表格 + 红章 + 手写签名的混合形态，OCR 难度低，数据密度极高。**先抽 1-3 页就够建立每份 PDF 的核心数据。**

### 3.2 原始记录页 4-N（待评估）

剩余 6-50+ 页推测是各检验项目的**原始记录**：

- 性状原始记录（描述性）
- 显微鉴别记录（带显微镜照片或绘图）
- 薄层鉴别记录（带 TLC 板照片）
- 水分测定原始记录（烘箱+电子天平读数）
- 灰分测定原始记录（炽灼+冷却+称量）
- 重金属测定原始记录（含 ICP/AAS 仪器输出？）
- 含量测定原始记录（HPLC 谱图 + 积分数据）
- 农残测定（GC-MS 谱图）

这些原始记录页**才是 F-001 二阶检验的核心战场**——因为：
- 仪器谱图反向计算的难度比"填表"高一个量级（需要伪造谱峰形态/积分面积/保留时间）
- HPLC 含量值如果不一致，要看是仪器误差、人工读数、还是有意调整
- TLC/显微等"主观判读"项目如果都"完美符合"，就是嫌疑信号

**建议策略**：v0.1 先只抓主页 1-3，建立每份 PDF 的核心索引；待 6 批同物料数据汇总后再针对**含量测定原始记录**（HPLC 谱图）做深入 OCR。

### 3.3 工作量估算

- **主页 1-3 × 81 PDF = 243 页**，约等于半个 batch 的工作量，1-2 个 session 可完成
- **完整原始记录 × 81 PDF ≈ 1500-3000 页**，需要 5-10 个 session

## 4. F-001 二阶检验初步判读（基于甘草饮片 L1 这一份）

### 4.1 单批数据"舒适过线"检查

| 指标 | 标准 | 实测 | %_LOD/LOQ | 解读 |
|------|------|------|-----------|------|
| 水分 | ≤12.0% | 8.1% | 67.5% LOD | 中间地带 |
| 总灰分 | ≤5.0% | 2.2% | 44% LOD | 离上限远 |
| 铅 | ≤5 mg/kg | 1 mg/kg | 20% LOD | 远 |
| 镉 | ≤1 mg/kg | 0.03 mg/kg | 3% LOD | 极远 |
| 砷 | ≤2 mg/kg | 0.1 mg/kg | 5% LOD | 远 |
| 汞 | ≤0.2 mg/kg | 0.02 mg/kg | 10% LOD | 远 |
| 铜 | ≤20 mg/kg | 4 mg/kg | 20% LOD | 远 |
| 药屑及杂质 | ≤3% | 1% | 33% LOD | 中间 |
| **甘草苷** | ≥0.45% | **0.87%** | **193% LOQ** | 超 LOQ 1 倍 |
| **甘草酸** | ≥1.8% | **2.5%** | **139% LOQ** | 超 LOQ 39% |

**单批初判**：数据**不像反向计算的"恰好过线"**。如果是反向计算填的，更可能写 0.46%（刚过 0.45%）而不是 0.87%。但单批不足以下结论。

### 4.2 6 批数据分布检验设计

待完成 6 批同物料 OCR 后，做以下三种检验：

1. **标准差分析**：同物料 6 批含量测定值的 σ；如果 σ < 0.05%（绝对值），可能反向计算；如果 σ > 0.10%，更可能是实际测量
2. **集中度分析**：6 批是否都集中在某个"舒适值"（如全在 0.85-0.90% 之间）；高集中度 = 反向计算嫌疑
3. **月份/季节趋势**：实际中药材含量受采收季、储存时间影响；如果 6 批含量与时间无关联，反向计算嫌疑加深

这 3 个分析可以**机械化**。Stage 4 跨批分析时直接套用。

### 4.3 v0.2 命题中的 layer 判定

来料检验数据**暂归 layer = "tentative_actual"**。最终归层取决于 §4.2 的 6 批分析结果：

- 若分布合理 → 升级为 `actual`（中层 / 实际工序执行的上游 ground truth）
- 若分布显示反向计算 → 降级为 `compliance`（与批生产记录同层）

## 5. Schema 扩展提议（v0.1 → v0.2 触发）

新增顶层 section：

```jsonc
"incoming_qc_来料检验": IncomingQCRecord[]
```

每条 IncomingQCRecord 字段已在 batch-record.json 中实例化（甘草饮片 L1），关键字段：

- 报告基本信息：report_id / 物料代码 / 进厂批号 / 厂家批号 / 供货单位 / 规格 / 来货数量 / 检验日期 / 报告日期
- 检验项目：性状 / 鉴别[] / 检查{水分/灰分/重金属/二氧化硫/杂质/农残} / 含量测定[]
- 每个数值字段必须含：标准 + 实测 + _pct_of_LOD（或 _倍率_of_LOQ）+ 结论
- 派生字段 `_F-001_indicators`：单批舒适过线判定 + 6 批分布判定策略
- 派生字段 `_lot_闭环`：批号 ↔ 批生产记录的精确映射 + 本批使用率
- 派生字段 `_layer_classification_v2`："tentative_actual" / "actual" / "compliance"
- 附属文档：物料放行单 / 请验单（form_id 不同但同样在 PDF 中）

完整 schema v0.2 文档将在累积 ~5 份不同物料 OCR 后正式起草，避免过早 lock 错误结构。
