# 03 多角色协作框架

> 本文件定义华典智谱以"中型开发团队"方式运作的角色体系、边界、交接规范。
> 核心原则：**每个角色在各自专业领域内拥有独占的决策权，其他角色不得越位替代。**
> 这不仅仅是组织形式——它也决定了 Claude Code 会话启动时要装入哪个 agent 身份，以避免"一个万能模型什么都做"导致的专业深度不足。

---

## 一、为什么要分角色

实践问题：
- **LLM 窗口限制** — 单线程塞满所有上下文会丢失细节
- **专业纵深不足** — 既懂古籍又懂 GraphQL 又懂 UI 的"通才模型"在每一面都不深
- **决策责任不清** — 当前端样式出问题，是工程师自己改还是找设计师？没有分工就没有质量
- **商业化合规** — 未来引入真人协作者时，AI 团队结构要能自然对接

## 三、职责矩阵（RACI 简化）

`R` = Responsible（执行方）；`A` = Accountable（最终责任方）；`C` = Consulted（咨询方）；`I` = Informed（知会方）

| 任务 | 架构 | PM | 历史 | 设计 | 后端 | 管线 | 前端 | QA | DevOps | 分析 |
|------|------|----|----|------|------|------|------|------|------|------|------|
| 架构决策 / ADR | **A/R** | C | C | I | C | C | I | I | C | I |
| PRD / 商业化取舍 | C | **A/R** | C | C | I | I | I | I | I | C |
| 实体归类 / 歧义仲裁 | C | I | **A/R** | I | C | C | I | C | I | I |
| 视觉 / 交互原型 | I | C | I | **A/R** | I | I | C | I | I | I |
| DB Schema / 迁移 | C | I | C | I | **A/R** | I | I | C | C | I |
| LLM Prompt / 管线 | C | I | C | I | I | **A/R** | I | C | I | I |
| 前端组件实现 | I | I | I | C | I | I | **A/R** | C | I | I |
| 测试用例 / 黄金集 | C | I | C | I | C | C | C | **A/R** | I | I |
| Docker / CI / 部署 | C | I | I | I | C | C | C | I | **A/R** | I |
| 埋点 / A/B / 看板 | C | C | I | C | C | I | C | I | I | **A/R** |
| 安全 / 合规审查 | **A** | C | I | I | C | C | C | C | R | C |
| 跨角色仲裁 | **A/R** | C | C | C | I | I | I | I | I | I |

规则：
- 一个任务必须有唯一 `A`（最终责任方）
- `R` 可以多个，但必须明确主导
- 任何决策变更必须通知所有 `A/R` 并更新 CHANGELOG

## 四、角色通用规范

### 4.1 工作启动（每个 agent 必须做）
```
1. Read CLAUDE.md（根）
2. Read docs/STATUS.md
3. Read 自己对应的 .claude/agents/{role}.md
4. Read 本次任务对应的 docs/tasks/T-NNN-*.md
5. 输出一个 TodoList，等用户/主导角色确认后再动手
```

### 4.2 工作结束（每个 agent 必须做）
```
1. 生成本任务的交付物摘要
2. 更新 docs/STATUS.md
3. 追加 docs/CHANGELOG.md 一条
4. 若产生新决策，追加 ADR
5. 若影响其他角色，在任务卡标注 "handoff_to: [role]"
```

### 4.3 越位禁区（违反视为严重违规）

| 角色 | 不得做 |
|------|--------|
| 架构师 | 不写业务代码；不做视觉设计 |
| PM | 不改 schema；不改代码；不决定技术栈 |
| 历史专家 | 不写代码；不改 schema 字段名（只能提议） |
| 设计师 | 不改数据结构；不写后端逻辑 |
| 后端工程师 | 不决定产品功能取舍；不改视觉；不独自定义 LLM prompt |
| 管线工程师 | 不决定实体定义（找历史专家）；不改前端 |
| 前端工程师 | 不决定视觉（找设计师）；不改 schema |
| QA | 不改架构决策；不改业务代码（可提 PR 建议） |
| DevOps | 不改业务代码（基建除外） |
| 分析师 | 不改数据模型；不改业务代码（埋点接入由前端/后端落实） |

### 4.4 冲突升级机制

```
两个角色意见分歧
    ↓
先在任务卡评论区用 "@{role}" 互相协商
    ↓ 仍冲突
升级到首席架构师仲裁（记入任务卡）
    ↓ 仲裁涉及架构宪法
升级到用户决策 + 修宪流程（docs/00）
```

## 五、交接物标准

### PM → 设计师
- 产出 PRD（`docs/prd/PRD-NNN-*.md`）含：目标用户、核心场景、成功指标、功能列表、不做清单
- 不含具体界面描述（那是设计师的工作）

### 设计师 → 前端工程师
- Figma 链接或 `docs/design/D-NNN-*.md`（若无 Figma，用 ASCII 线框 + 组件规范）
- 明确：组件名、props 清单、状态图、交互反馈、空状态、错误态
- 不含具体代码实现

### 历史专家 → 管线工程师
- `data/dictionaries/*.yaml`（字典文件）
- `docs/historical_rules/*.md`（实体规则、消歧规则、典故清单）
- 不直接改代码；但可用 markdown 表格提供"此实体应如何抽取"的样本

### 架构师 → 后端 / 管线 / 前端
- `docs/decisions/ADR-NNN-*.md`
- Schema 骨架（见 docs/02）
- 不写实现代码

### 后端 → 前端
- GraphQL schema 文件（SDL）
- 示例 query / mutation
- 类型通过 `packages/shared-types` 共享

### 管线 → 后端 + QA
- 入库的实体记录（由管线负责写入 DB）
- 黄金集样本（`data/golden/*.json`）
- TraceGuard 质检报告

### QA → 所有人
- 测试覆盖率报告
- 质检规则清单（`docs/qc/rules.md`）
- 失败用例定位 + 责任角色指派

### 分析师 → PM + 架构师
- 周度用户数据看板（Metabase / PostHog Dashboard）
- 月度洞察报告（`docs/analytics/insight-YYYY-MM.md`）
- A/B 实验结论

## 六、Claude Code 用法

### 在 Claude Code 中调用某角色

```bash
# 方式 1: 启动会话时在 prompt 中声明
claude
> 你现在担任 {role}。Read .claude/agents/{role}.md 并按其规范工作。本次任务 ID: T-NNN
```

或者，将来 Claude Code 支持 subagent 时：
```bash
# 方式 2: 使用 subagent（未来）
/agent historian
> 帮我评审《史记·项羽本纪》的实体抽取结果
```

### 跨角色协作模板（在单次 Claude Code 会话中）

```
你当前扮演 {主导角色}。本任务需要向 {其他角色} 咨询/获得交付物。
1. 完成你的主导工作
2. 明确指出需要 {其他角色} 提供什么
3. 如果需要切换身份，提示我 "请切换到 {其他角色} 视角继续"
不要擅自扮演你不是的角色做决定。
```

## 七、扩展与未来

当前是 AI agent 化团队。未来引入真人协作时：
- 真人可以接管任意一个角色
- AI agent 作为该角色的"初稿生成器"和"评审助手"
- 文档结构（ADR / 任务卡 / STATUS / CHANGELOG）对 AI 和真人都友好

当团队扩大时，可按 squad 分组：
- 数据 squad：历史专家 + 管线 + QA
- 产品 squad：PM + 设计师 + 分析师
- 工程 squad：后端 + 前端 + DevOps
- 跨 squad：架构师

---

## 八、本框架的演进

本框架本身也是活文档。当：
- 角色边界频繁被越过 → 说明边界定义不清，修订边界
- 某类决策反复升级到架构师 → 说明需要新增角色或重新分工
- 任务卡大量出现某种标签 → 考虑是否需要专职角色

每次修订记入 CHANGELOG，重大修订需 ADR。
