## 0. 重要前言：本文档与 v1.0 的关系

v1.0 (`docs/archive/华典智谱_架构设计文档_v1.0.md`) 基于 **C-route**（C 端古籍知识平台）定位起草。本 v2.0 基于 **D-route**（Agentic KE 框架 + 案例参考实现）定位重写，是项目的**当前** authoritative 架构文档。

v2.0 与 v1.0 的关键差异：

| 维度 | v1.0 (C-route) | v2.0 (D-route) |
|------|----------------|----------------|
| 项目身份 | 古籍 AI 知识平台（产品）| Agentic KE 框架 + 史记参考实现（方法论 + 工具）|
| 架构组织 | 4 层（数据 / 知识 / 应用 / 体验）| 双轴（框架层 + 案例实现层）|
| 核心交付 | 完整产品（Web / Mobile / API）| 框架代码 + 方法论文档 + 案例实现 |
| 数据规模目标 | 史记 130 篇全本 → 二十四史 → ... | 史记 5-10 篇典型章节（案例够用即停）|
| 用户群 | C 端历史爱好者 / 学生 | KE 工程团队 / 学者 / AI agent 项目作者 |
| 商业模式 | C 端订阅 + 教育版 + 文旅 | 开源 + 长期机会主义（咨询/培训/B2B）|

**v1.0 仍然作为档案保留**，其早期数据模型设计、NER 范畴、ADR 决策等大部分仍然在 v2.0 中保留（详见 §4 案例实现层）。

## 1. 设计哲学（D-route 核心）

### 1.1 一句话总结

> **"用 AI 让做事变得几乎免费的时代，'做对的事'和'团队做对的事'反而成为最稀缺的能力。"**

我们的产品是**让团队能做对的事**的工程框架，史记知识库是**这个框架真的 work** 的证明。

### 1.2 4 个根基判断

| # | 根基判断 | 推论 |
|---|---------|------|
| 1 | 单人 + AI 能高产，但不能扩展 | 需要团队级工程纪律 → role / sprint / ADR / invariant |
| 2 | 知识工程没有"Scrum"——这是空白 | 我们做这个空白象限，不在 LLM 应用 / RAG 类工具拥挤的赛道里卷 |
| 3 | 案例验证 > 凭空设计 | 框架抽象**必须**来自真实代码，不允许"先设计再找案例" |
| 4 | 写作低频但有节律 | 用户不擅长写作 → AI 协作 + 月/季度软节律 + 代码即文档 |

### 1.3 不可变原则（项目宪法）

详见 [docs/00_项目宪法.md](docs/00_项目宪法.md) C-1 ~ C-25。其中 D-route 关键 4 条：

- **C-22 案例服务于框架** — sprint 工作必须先回答"对框架抽象的价值"
- **C-23 工程模式必须可抽象** — 任何 invariant / guard / pattern 设计时标注 domain-specific 参数
- **C-24 写作低频但有节律** — 月度决策日记 + 季度方法论文章
- **C-25 互补不竞争** — 与 shiji-kb / 字节识典古籍 / Anthropic 工具等公开互补关系

### 2.1 双轴边界规则（关键）

任何代码 / 文档 / 决策都要明确归属哪一轴。原则：

- **框架层** ←接收← 通用抽象、领域无关 pattern、多领域可复用工程模式
- **案例实现层** ←接收← 史记 domain knowledge、人物/朝代/事件领域数据、NER prompt 调优、特定 schema 字段

**抽象提升路径（C-23）**：当某个案例层的 pattern 在 ≥2 个不同 surface（章节 / 实体类型 / 朝代）上复用，且其核心逻辑可参数化为领域无关时，应抽出到框架层。

举例：
- ✅ `evaluate_pair_guards(rule, GUARD_CHAINS)` 是框架层 — 接受任意 guard 链
- ✅ `cross_dynasty_guard(200yr)` 在两轴交界 — 算法领域无关，参数（200yr 阈值 / 朝代列表）领域特定
- ✅ `dynasty-periods.yaml` 是案例层 — 中国古代朝代年表
- ❌ "古人模拟器" 不在任何一轴里 — 是 sunset 的 C 端产品方向

### 3.4 框架与案例的契约

框架层提供给案例层的接口（草案）：

```python
# 伪代码示意
from huadian_framework.identity import resolve, GuardChain
from huadian_framework.invariants import Invariant, run_all
from huadian_framework.audit import log_decision

# 案例层注入领域 domain
chain = GuardChain([
    cross_period_guard(threshold_years=200),  # 领域参数：200 年来自史记
    state_prefix_guard(states_yaml="data/states.yaml"),
])
result = resolve(person_a, person_b, rule="R1", chains={"R1": chain})
```

## 6. 演化路径

### 6.1 v2.0 → v3.0 触发条件

本文档（v2.0）是 D-route 转型即时产物，预计随框架抽象成熟会在 6-12 个月内更新。触发 v3.0 的条件：

1. Sprint L 完成首批框架抽象 → §3 框架层架构有真实代码可写
2. 第 2 个跨领域案例（佛经 / 法律 / 医疗 / 等）启动 → §4 案例层需扩展为多案例视角
3. 框架命名（ADR-030）+ 仓库结构（单仓 / 分仓决策）有定论
4. Layer 4 商业化探索启动 → 增加商业架构章节

### 6.2 v2.0 当前缺陷（ack）

诚实承认本版本的不完整：

- §3 框架层很多还是规划，没有真实代码引用 — 待 Sprint L 充实
- §4 案例层细节大量在 v1.0 + 各 ADR + 各 sprint logs 中，本版未全部 inline — 需要时引用即可
- 没有 schema 详细字段表 — 这部分稳定（v1.0 + 各 migration 已经定义），不在 v2.0 重复
- 没有 API 详细 endpoint 表 — 同上

这些"不完整"是有意的：v2.0 的优先级是**把双轴架构概念立住**，细节随 Sprint 推进自然填充。
